🎯 1. 为什么需要协同度量体系

"感觉团队合作'还不错'的项目，往往在 Beta 阶段才发现协同效率已经烂到骨子里——因为没人在量化地看数据。你不度量的东西，你就无法改善。"

— 某 10 年经验游戏制作人

1.1 协同失效的代价

失效维度典型症状隐性成本
需求流转策划提的需求 2 周后美术才看到每延误 1 天 ≈ 浪费全团队 0.5 人日
审核阻塞资产做完后等审核 5 天没人看Pipeline 堵塞 → 批量延期 → 版本滑坡
Bug 互踢美术说不是Bug，QA说是Bug，来回 3 轮单 Bug 扯皮成本 = 直接修复的 4 倍
返工循环资产反复修改超过 3 轮还不合格单资产成本翻 3 倍 + 士气损耗
信息黑洞改了东西没通知下游，引擎里的还是旧版Bug 修复后重复出现 → 信任崩塌

1.2 度量体系的价值

  • 早期预警：在问题恶化为"事故"之前通过数据发现苗头
  • 客观决策：用数据代替"感觉"，避免协同问题归因于个人
  • 持续改进：每个迭代/Sprint 可对比改善幅度
  • 资源优化：识别瓶颈环节，精准投入人力/工具

🧭 2. 协同健康度四象限模型

象限关注维度核心问题关键指标
📈 效率象限流程速度"多快能交付？"需求流转周期 · 审核响应时效 · 阻塞时长
⭐ 品质象限产出质量"产出够好吗？"一次通过率 · Bug 密度 · 返工轮次
🤝 协作象限跨部门摩擦"合作顺畅吗？"变更率 · 阻塞互等时长 · 信息同步覆盖率
😊 健康象限团队状态"人还好吗？"加班指数 · 流失率 · 满意度
⚠️ 制作人注意：四个象限需要平衡看。只追求效率不看品质 = 堆垃圾；只看品质不管效率 = 项目失控。如果健康象限持续红色，其他三个象限的数据都不可信（人快崩了的时候数字是假的）。

6.2 数据可视化建议

  • 实时看板：Grafana / 飞书多维表格 / 自研 Dashboard，挂在团队工区大屏
  • 周报自动生成：Python 脚本定时从各系统 API 拉取数据 → 生成 Markdown/PDF
  • 趋势对比：每个指标展示近 8 周折线图 + 基线参考线
  • 红绿灯配色：健康=绿、警戒=黄、红线=红，一眼可判

📐 7. 阶段性基线（预研/Alpha/Beta/上线）

💡 核心思路：不同阶段容忍度不同。预研期允许"混沌"，但越接近上线，指标越要严格。
指标预研期AlphaBetaRC/上线
一次通过率≥ 50%≥ 60%≥ 70%≥ 80%
需求变更率≤ 40%（允许探索）≤ 25%≤ 15%≤ 5%
A/S Bug 遗留不限≤ 10≤ 30
审核响应达标率≥ 70%≥ 80%≥ 90%≥ 95%
性能达标率≥ 60%≥ 80%≥ 95%100%
外包交付合格率≥ 60%≥ 70%≥ 75%≥ 85%
周均加班指数≤ 10h≤ 10h≤ 15h（冲刺）≤ 20h（短期）
🚨 红线：即使是"冲刺阶段"，周均加班 > 20h 持续超过 4 周 = 管理失败。此时应该砍功能而不是继续加班。

🔄 8. 度量驱动的持续改进循环

8.1 PDCA 改进循环

阶段动作产出负责人
Plan（计划）分析上一周期数据 → 定位最差的 2~3 个指标 → 制定改善 Action改进计划表APM
Do（执行）在下一个 Sprint 内落地改进措施执行记录相关组
Check（检查）Sprint 结束后对比数据是否改善数据对比报告APM
Act（固化）有效的措施写入 SOP，无效的复盘后调整方向SOP 更新APM + 制作人

8.2 Sprint 复盘聚焦模板

复盘项内容
本 Sprint 最大的协同瓶颈是什么？用数据说明（哪个指标、差多少）
根因分析（5 Why）连问 5 个"为什么"找到系统性原因
已尝试的措施是否有效？用数据对比前后变化
下一 Sprint 的 1~2 个改进 ActionSMART 原则：具体/可衡量/可实现/相关/有时限
需要制作人决策的事项超出 APM 权限的调整（加人/换人/砍需求）

8.3 度量成熟度阶段

成熟度特征典型团队画像
L1 无度量完全靠感觉管理，问题靠"吵架"暴露10 人以下初创团队
L2 事后统计月末手动统计数据，滞后 2~4 周大多数中型团队现状
L3 周期监控每周自动生成报告 + 预警机制管理成熟的 50+ 人团队
L4 实时驱动实时 Dashboard + 自动预警 + 改进闭环顶尖工作室标杆
💡 建议：大多数团队应以 L3（周期监控）为目标。从 L1 到 L3 通常需要 2~3 个月的工具搭建和流程磨合。不要一上来就追求 L4，先把周报跑起来。
