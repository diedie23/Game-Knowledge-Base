### 1.2 APM 视角关注重点

| 关注点 | 具体内容 | 优先级 |
|:---:|:---:|:---:|
| **风格一致性** | 所有角色/元素遵循同一色板和描边规范，多CP并行时尤其重要 | 🔴 高 |
| **分辨率统一** | 画布尺寸 2^N，不同部件基准一致 | 🔴 高 |
| **动画帧率** | 像素风 8~12FPS，高清风 12~24FPS，全项目统一 | 🟡 中 |
| **图集效率** | 合理规划 Atlas，控制 Draw Call | 🟡 中 |
| **迭代管控** | 设计稿审批→制作→内审→修改，每环节不超过2轮 | 🟡 中 |
| **资源命名** | 统一命名规范，CI/CD 自动校验 | 🟢 低 |

### 1.3 外包管理难点和常见风险

| 风险 | 描述 | 应对措施 |
|:---:|:---:|:---:|
| 🔴 **风格偏差** | 不同CP画风差异大，玩家能感知到"拼凑感" | 提供完整风格参考包（3~5张标杆图+色板+描边规范）；首批交付前要求2张测试稿对齐 |
| 🔴 **分辨率混乱** | 部分CP不遵守2^N，导致引擎压缩异常 | 提供预检工具链接，要求交付前自检 |
| 🟡 **动画不兼容** | 帧动画尺寸不一致，骨骼绑定方式不统一 | 提供骨骼模板文件+动画参考视频 |
| 🟡 **命名不规范** | 中文文件名、空格、大小写混乱 | 在需求文档中附命名规范+批量重命名工具 |
| 🟡 **交付延期** | 小CP团队人力波动大 | 关键路径资源分配给2家CP，互为备份 |
| 🟢 **过度设计** | 休闲游戏不需要过于复杂的细节 | 明确"够用标准"，避免CP为了充工时做过多细节 |

### 1.5 验收标准建议

| 检查项 | 合格标准 | 验收方式 |
|:---:|:---:|:---:|
| 画布尺寸 | 2^N（如512×512） | 工具自动检测 |
| 文件格式 | PNG（带Alpha） | 工具自动检测 |
| 色彩一致性 | 与色板偏差 ΔE < 5 | 肉眼+取色器 |
| 描边规范 | 统一粗细和颜色 | 肉眼对比标杆图 |
| 动画流畅度 | 无卡顿/跳帧 | 引擎内播放 |
| 骨骼兼容性 | 导入引擎无报错 | 引擎导入测试 |
| 命名规范 | 符合命名规则 | 脚本自动校验 |
| 文件大小 | 单图 ≤ 1MB | 工具自动检测 |
| 透明通道 | 无残留半透明像素 | 工具检测 |
| 衔接对齐 | 部件衔接处无缝隙 | 引擎内叠加测试 |

### 2.2 APM 视角关注重点

| 关注点 | 具体内容 | 优先级 |
|:---:|:---:|:---:|
| **面数预算** | 每个资源类型的面数上限必须在制作前明确，不可后期砍面 | 🔴 高 |
| **贴图规格** | 贴图尺寸/张数/通道用途，需在TA手册中明确 | 🔴 高 |
| **PBR 标准** | Metallic/Roughness 工作流 vs Specular 工作流，全项目统一 | 🔴 高 |
| **LOD 分级** | LOD0~LOD3 的面数比例和切换距离 | 🟡 中 |
| **骨骼规范** | 骨骼数量上限、命名规范、IK 链定义 | 🟡 中 |
| **动画规范** | 动画片段时长、过渡帧数、根运动设置 | 🟡 中 |
| **外包分工** | 高模/低模/贴图/动画可分包给不同CP | 🟡 中 |
| **版本管理** | 3D资源的版本管理（Perforce/Git LFS） | 🟢 低 |

### 2.3 外包管理难点和常见风险

| 风险 | 描述 | 应对措施 |
|:---:|:---:|:---:|
| 🔴 **面数超标** | CP为了效果堆面数，导入引擎后严重卡顿 | 制作前签署面数预算表，分阶段检查点（高模完成/低模完成各验一次） |
| 🔴 **UV重叠/浪费** | UV利用率低于60%，贴图精度不够 | 要求UV利用率 ≥ 75%，提供UV检查工具 |
| 🔴 **PBR参数错误** | 金属度/粗糙度范围不对，在不同光照下表现异常 | 提供PBR参数参考表 + 材质球测试场景 |
| 🟡 **高低模不匹配** | 低模拓扑与高模形态差异大，法线烘焙出现伪影 | 要求提交低模时附带高模↔低模对比截图 |
| 🟡 **骨骼不兼容** | 骨骼命名/层级/数量与项目标准不一致 | 提供标准骨骼模板，要求在模板基础上制作 |
| 🟡 **动画脚滑/穿模** | 角色脚底不贴地、武器穿透身体 | 要求提交引擎内录屏（非DCC软件截图） |
| 🟡 **交付格式混乱** | FBX版本不对、缺少必要通道贴图 | 提供标准交付包模板+自动检查脚本 |
| 🟢 **LOD制作敷衍** | LOD减面不均匀，远景出现穿帮 | 要求LOD层级截图+面数表 |

### 2.5 验收标准建议

| 检查项 | 合格标准 | 验收方式 |
|:---:|:---:|:---:|
| 面数 | 角色 ≤ 30,000面（LOD0）| 引擎Stats面板 |
| 贴图尺寸 | 身体2048，配件1024，总显存 ≤ 20MB | 引擎资源审计 |
| UV利用率 | ≥ 75% | UV Layout截图 |
| 法线烘焙 | 无明显接缝/伪影 | 肉眼+引擎多角度旋转 |
| PBR参数 | Metallic 0或1（非金属/金属），Roughness 0.1~0.9 | 材质球测试场景 |
| 骨骼数量 | ≤ 75个（移动端）/ ≤ 150个（PC端） | 引擎Skeleton面板 |
| 动画帧率 | 30 FPS | DCC+引擎内验证 |
| 穿模检测 | 所有动画无穿模 | 引擎内逐动画播放 |
| LOD切换 | 各LOD切换平滑，无明显跳变 | 引擎内远近拉伸测试 |
| 导入兼容 | FBX导入无报错/警告 | 引擎Import Log |

| **Live2D 流畅度** | 待机动画自然度、呼吸感、头发/衣物物理效果 | 🟡 中 |
| **审核合规** | 服装暴露度、敏感元素、文化合规（出海项目尤其重要） | 🔴 高 |
| **卡通渲染一致性** | Toon Shader参数统一，不同角色在同场景下表现一致 | 🟡 中 |
| **版本管理** | 立绘多版本管理（审核版/海外版/国内版） | 🟡 中 |

### 3.3 外包管理难点和常见风险

| 风险 | 描述 | 应对措施 |
|:---:|:---:|:---:|
| 🔴 **立绘品质波动** | 二次元画师个人风格强，不同画师差异巨大 | 严格试稿制度（线稿→上色→完成 三阶段审核）；核心角色只分配给A级CP |
| 🔴 **IP/版权问题** | 联动角色使用授权范围不清，CP可能参考过重 | 法务前置介入，提供"参考禁区"清单 |
| 🔴 **审核退回** | 服装/姿态不符合审核标准 | 制作前提供"安全线"示例图，减少返工 |
| 🟡 **表情差分不一致** | 不同表情的五官比例偏移 | 要求在同一PSD中制作，叠加对比检查 |
| 🟡 **Live2D 拆分错误** | 图层拆分粒度不对，导致动态效果生硬 | 提供拆分模板+优秀案例参考 |
| 🟡 **Toon Shader适配** | 外包制作的3D模型导入后卡通渲染效果与原画差距大 | 提供Shader参数预设+标准测试场景 |
| 🟢 **多版本管理混乱** | 国内版/海外版/审核版 文件混淆 | 建立版本分支命名规范（如 _CN / _GL / _REVIEW） |

### 3.5 验收标准建议

| 色彩还原 | 与审批稿色彩偏差 ΔE < 3 | 取色器对比 |
| Live2D流畅度 | 待机动画自然，无穿模/抖动 | 编辑器播放+引擎内播放 |
| 3D卡通渲染 | 与原画相似度 ≥ 80%（主观评估+参考对比） | 标准测试场景截图对比 |

| 图层规范 | PSD图层命名规范，分组清晰 | 人工检查图层面板 |

| 分辨率适配 | 移动端测试无模糊/锯齿 | 真机截图 |
| 性能指标 | Live2D ≤ 3ms/帧，3D角色 ≤ 20,000面 | 引擎Profiler |

## 四、三类游戏对比总结

### 4.1 核心差异对照

| 维度 | 轻度2D休闲 | 重度3D游戏 | 二次元手游 |
|:---:|:---:|:---:|:---:|
| **核心资源** | 2D角色/场景元素 | 3D模型/PBR材质 | 立绘/Live2D/卡通渲染 |
| **制作周期** | 短（2~4周/角色） | 长（4~10周/角色） | 中长（5~12周/角色） |
| **技术门槛** | 低 | 高 | 中（但艺术门槛高） |
| **外包依赖度** | 高（批量产出） | 高（专业分工） | 高（画师资源稀缺） |
| **品质敏感度** | 中（风格统一优先） | 高（技术指标硬性） | 极高（玩家审美苛刻） |
| **最大风险** | 风格不一致 | 面数/性能超标 | 立绘品质波动 |
| **APM核心能力** | 风格管控+批量管理 | 技术规范+分工协调 | 画师资源+审核合规 |

### 4.2 APM 通用管理建议

1. **前置化规范**：所有规范在项目启动时制定，而非出问题后补救
2. **工具化检测**：能用工具自动检测的指标，不要用肉眼
3. **分阶段验收**：不要等完成后才验收，关键节点必须检查
4. **模板化沟通**：所有需求用模板下发，减少理解偏差
5. **冗余规划**：关键资源至少2家CP备份，避免单点故障
6. **数据化评估**：建立CP评分体系（交付质量/按时率/返工率），用数据而非感觉选CP
