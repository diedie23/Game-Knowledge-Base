💥 踩坑 1：外包翻车 — 首批角色交付全部打回

高频"按你们给的参考图做的呀？" — 外包启动阶段的经典翻车

🎭 典型场景还原

👔
APM："签了 CP-Alpha，下发 5 个角色建模需求，提供参考图和简要口头说明。"
📋
外包 CP："按照你们给的参考图做的呀？"
🎨
主美："风格偏差大，面数超标，UV 展开不规范，4 个全部打回！"
👔
APM：双方扯皮 2 周，整体延期 18 天……

🧪 问题根因拆解

层级原因谁的锅
表面CP 交付质量差，风格偏差大外包 CP 执行偏差
中层只给了参考图和口头说明，没有详细风格指南APM + 主美没提供标准文档
深层外包启动流程中没有"风格指南交付"这个强制节点流程缺陷：启动 Checklist 缺失

🔑 APM 破局思路

① 外包启动前必须提供《风格指南》文档（≥ 10 页）

含色板、材质标杆、UV 标准、面数红线，不能只给参考图。关键原则："参考图只是方向，风格指南才是标准。"

② 推行「首件样品机制」

CP 先做 1 个标杆件，通过主美审核后再批量生产。避免批量翻车，把风险控制在第一个角色。

③ 每周 2 次在线 Review

问题及时纠正，不要等交付才发现偏差。APM 排期中预留 Review 时间，不是额外负担而是防翻车保险。

④ 合同增加约束条款 + 主美做标杆角色

首批样品不通过可解约，倒逼 CP 重视首件质量。同时主美亲自做一个标杆角色作为 CP 的参照标准，消除理解偏差。

❌ 翻车模式
给参考图 + 口头说明 → CP 凭感觉做 → 交付时风格偏差大 → 全部打回 → 扯皮 2 周 → 延期 18 天
➡️
✅ 正确模式
风格指南 + 标杆件 → 首件审核通过 → 批量生产 → 每周 Review 纠偏 → 交付一次过
⚡ APM 金句："参考图只是方向，风格指南才是标准。没有标准就开工，翻车只是时间问题。"

📅 踩坑 2：排期爆雷 — 场景组集体加班 3 周

高频"方案改了，部分区域重新布局" — 方案未锁定就量产的灾难

🎭 典型场景还原

📝
策划："主城方案 v1 完成，开始白盒吧。"
🎨
场景组：（埋头做了 2 周白盒）
📝
策划："方案改了，部分区域重新布局。"
🎨
场景组：😱 推倒重做 40%...
👔
制作人：（3 周后才看）"入口区域动线不对，需要改。"
🎨
场景组：💀 连续加班 3 周，2 人提出调岗……

🧪 问题根因拆解

层级原因谁的锅
表面策划方案频繁变更，场景组反复返工策划变更过于频繁
中层策划方案未经制作人确认就下发给美术策划 + APM 没做 Gate Review
深层方案评审流程缺少"制作人看白盒"强制节点；大型场景未拆分为模块化区域分批确认项目没有阶段 Gate Review

🔑 APM 破局思路

① 白盒阶段三方确认机制

白盒必须经制作人 + 策划 + 主美三方确认后才能进入美化阶段。APM 组织评审会，留存签字记录。关键原则："没有三方签字，不开工。"

② 方案变更走正式流程

任何方案变更必须走变更申请单：变更描述 → 影响面评估（人天、延期） → 策划主管审批。评估影响面后再执行，不接受口头变更。

③ 大型场景模块化拆分

按区域分批确认，降低一次性返工的风险。每个区域独立评审、独立冻结，局部修改不影响全局。

④ 排期预留缓冲 + 三阶段制度

排期预留 15~20% 缓冲 buffer 应对不可避免的需求变更。推行「设计评审 → 冻结 → 生产」三阶段制度：冻结后除致命问题外不接受变更。

❌ 爆雷模式
策划口头说开工 → 埋头做 2 周 → 方案改了 → 推倒 40% → 制作人 3 周后才看 → 又改 → 加班 3 周
➡️
✅ 正确模式
方案评审 → 三方签字确认 → 冻结 → 模块化生产 → 变更走申请单 → 有序推进
⚡ APM 金句："方案没锁定就开工，等于在流沙上盖房子。评审不到位，加班到崩溃。"

🔧 踩坑 3：资产返工 — 角色面数超标被 QA 批量打回

高频"为什么中间没检查？" — 制作过程中缺乏自动化卡点

🎭 典型场景还原

🎨
建模师：（精雕细琢每个角色，追求极致细节）
🔧
TA 脚本：⚠️ 面数检查 → 8 个角色超标（红线 15k，实际 18k~22k）
🎨
角色组长："为什么中间没检查？"
🔧
TA："自动化检查需求一直排不上优先级……"

🧪 问题根因拆解

层级原因谁的锅
表面8 个角色面数超标，被 QA 批量打回建模师追求细节逐渐加面
中层面数检查只在最终走查时做，制作过程中无检查APM + TA 没搭建中期检查
深层自动化面数检查工具优先级被低估，未开发；建模师无实时提醒流程 + 工具缺陷

🔑 APM 破局思路

① 搭建 Pre-commit 面数检查 CI

超标自动拒绝提交，从源头卡住问题。集成到 Jenkins/GitHub Actions，每次提交自动校验面数、贴图格式、命名规范。

② 每 Sprint 中期做一次面数抽检

不等到最终走查才发现问题。APM 在 Sprint 中期组织 TA 做关键资产抽检，早发现 = 低成本修复。

③ 在任务卡片上标注面数红线

让建模师时刻有数据意识，心中有数。每个任务卡标注：
面数上限：15,000 · 贴图：1024×1024 · 骨骼数：30

④ TA 提供自动化减面脚本 + 组长审核

TA 提供自动化减面脚本辅助快速减面，降低返工成本。建模组长逐个审核减面方案，优先减裙摆背面、鞋底、不可见装饰区域。

❌ 返工模式
建模师追求细节 → 过程无检查 → 最终走查发现超标 → 8 个角色批量打回 → 集中返工减面
➡️
✅ 正确模式
任务卡标红线 → Pre-commit CI 自动拦截 → Sprint 中期抽检 → 实时纠偏 → 一次过
⚡ APM 金句："质量问题发现得越晚，修复成本越高。最后走查才发现超标，已经是最贵的时候了。"

⚡ 补充踩坑：3 个隐性生产雷区

雷区典型台词真正原因APM 对策
📋 启动缺陷 "不是给了参考图吗？" 外包启动只靠口头 + 参考图，没有标准文档 制定《外包启动 Checklist》：风格指南 + 技术规范 + 首件机制，缺一不可
🔄 变更失控 "老板说改就改呗" 需求变更无评估、无审批、无冻结期 建立变更评估单 + 冻结机制：影响 > 2 天的变更需主管审批，Beta 前 1 月冻结
🔍 检查缺位 "自动化检查一直排不上" 质量检查只在最终走查，过程中无卡点 推动 TA 搭建 CI 自动检查，APM 确保自动化工具优先级不被挤压

🏗️ APM 踩坑预防体系框架

总结以上所有踩坑案例，APM 应在项目中搭建以下5 道防线：

防线措施工具/形式频率
第 1 道：规范先行立项期制定《美术资源规范》《外包启动 Checklist》《风格指南模板》文档模板立项期
第 2 道：阶段门禁方案评审 → 白盒确认（三方签字）→ 美化 → 终审评审会议每个阶段
第 3 道：自动化卡点CI 流水线集成：面数/贴图格式/命名规范自动校验Jenkins/GitHub Actions每次提交
第 4 道：中期抽检每 Sprint 中期对关键资产做抽检Checklist + 抽检记录每 Sprint
第 5 道：变更管控需求变更走正式流程，Beta 前 1 月冻结TAPD/飞书审批按需触发
💡 终极心法：事故本身不可怕，可怕的是同一个坑踩两次。每次事故复盘后，确保修复措施固化为流程/自动化，而不是"口头承诺下次注意"。
