## 案例 1：外包翻车 — 首批角色交付全部打回

### 【事故标题】
外包 CP-Alpha 首批 5 个角色建模交付，4 个不达标被打回

### 【发生阶段】
量产期（Alpha → Beta 阶段）

### 【严重等级】
🟠 **A 级**

### 【事故经过】

| 时间 | 事件 |
|------|------|
| 03-01 | 与 CP-Alpha 签约，下发 5 个角色建模需求 |
| 03-05 | 提供参考图和简要口头说明 |
| 03-20 | CP 提交第一批样品，内部 Review |
| 03-21 | 主美反馈：风格偏差大，面数超标，UV 展开不规范 |
| 03-22 | 打回 4 个，要求重做 |
| 03-25 | CP 反馈："按照你们给的参考做的"，双方扯皮 |
| 04-01 | 重做后交付，2 个通过，2 个仍有问题 |
| 04-08 | 第三轮修改后全部通过，整体延期 18 天 |

### 【根因分析 — 5-Why】

```
Why-1: 为什么首批交付质量差？
  → CP 对项目美术风格理解有偏差

Why-2: 为什么理解偏差？
  → 只给了参考图和口头说明，没有详细的风格指南文档

Why-3: 为什么没有风格指南？
  → 赶进度，认为"看参考图就行了"

Why-4: 为什么赶进度？
  → 风格指南编写未列入正式排期

Why-5: 为什么没排期？
  → 外包启动流程中没有"风格指南交付"这个强制节点
```

**🎯 根因**：外包启动流程缺少标准化的风格指南交付环节

### 【影响范围】

| 维度 | 影响 |
|------|------|
| 工期 | 延期 18 天，影响 Alpha 里程碑 |
| 成本 | 额外修改费用 ¥15,000 + 内部人力 10 人天 |
| 质量 | 2 个角色仍有细节瑕疵（B 级 Bug） |
| 士气 | 团队对外包质量信心下降 |

### 【解决方案】
1. APM 紧急介入，每日跟进 CP 修改进度
2. 主美亲自做了 1 个标杆角色作为 CP 参照
3. 剩余 2 个高风险角色改为内部承接

### 【预防措施】
1. ✅ 外包启动前必须提供《风格指南》文档（≥ 10 页）
2. ✅ 首件样品机制：CP 先做 1 个标杆件，通过后再批量
3. ✅ 每周 2 次在线 Review 进度
4. ✅ 合同中增加"首批样品不通过可解约"条款

### 【经验金句】
> 💣 **"给外包的规范永远比你想象的需要更详细。口头沟通 ≠ 有效沟通。"**

## 案例 2：排期爆雷 — 场景组集体加班 3 周

### 【事故标题】
主城场景因策划反复改方案导致排期爆雷，场景组连续加班 3 周

### 【发生阶段】
量产期（Alpha 后期）

### 【严重等级】
🔴 **S 级**

### 【事故经过】

| 时间 | 事件 |
|------|------|
| 02-15 | 策划提交主城设计方案 v1，场景组开始白盒制作 |
| 03-01 | 策划反馈："方案改了，部分区域重新布局" |
| 03-05 | 场景组白盒推倒重做 40%，排期后移 1 周 |
| 03-15 | 进入美化阶段 |
| 03-22 | 制作人审完说："入口区域动线不对，需要改" |
| 03-25 | 再次返工入口区域，排期后移 1 周 |
| 04-01~04-21 | 场景组连续加班 3 周赶进度 |
| 04-22 | 主城勉强完成，质量未达理想状态 |

### 【根因分析 — 5-Why】

```
Why-1: 为什么排期爆雷？
  → 场景方案变更 2 次，每次返工 1 周

Why-2: 为什么方案变更 2 次？
  → 策划方案未经制作人确认就下发

Why-3: 为什么没确认？
  → 策划认为方案已内部对齐

Why-4: 为什么实际没对齐？
  → 制作人忙，只看了文字描述，没看 3D 白盒

Why-5: 为什么没看白盒？
  → 方案评审流程中没有"制作人看白盒"这个强制节点
```

**🎯 根因**：场景方案评审流程不完善，缺少制作人对白盒的确认环节

### 【影响范围】

| 维度 | 影响 |
|------|------|
| 工期 | 场景延期 2 周，影响后续 UI 配合 |
| 成本 | 加班成本约 ¥40,000 |
| 质量 | 赶工导致部分区域完成度不足 |
| 士气 | 场景组疲惫，2 人提出调岗意向 |

### 【解决方案】
1. 场景组分两班倒，关键区域优先
2. APM 协调 2 名外包场景支援
3. 低优先级区域降低美化标准

### 【预防措施】
1. ✅ 白盒阶段必须经制作人 + 策划 + 主美三方确认后才能进入美化
2. ✅ 方案变更走正式变更流程，评估影响后再执行
3. ✅ 大型场景拆分为模块化区域，分批确认

### 【经验金句】
> 💣 **"没有制作人签字的方案，就不是最终方案。白盒确认是最便宜的纠错机会。"**

## 案例 3：资产返工 — 角色面数超标被 QA 批量打回

### 【事故标题】
Beta 版本走查发现 8 个角色面数超标，需集中返工减面

### 【发生阶段】
测试期（Beta 走查）

### 【严重等级】
🟠 **A 级**

### 【事故经过】

| 时间 | 事件 |
|------|------|
| 05-25 | Beta 版本提交，QA 开始走查 |
| 05-26 | TA 运行面数检查脚本，发现 8 个角色面数超标（15k 红线，实际 18k~22k） |
| 05-27 | APM 紧急召开会议，安排减面计划 |
| 05-28~06-05 | 角色组集中减面 + 重新检查 UV |
| 06-06 | 全部角色面数达标 |

### 【根因分析 — 5-Why】

```
Why-1: 为什么 8 个角色面数超标？
  → 制作过程中美术追求细节，逐渐加面

Why-2: 为什么没有及时发现？
  → 面数检查只在最终走查时做

Why-3: 为什么中间不检查？
  → 没有自动化的面数检查 CI

Why-4: 为什么没有自动化检查？
  → TA 一直在忙其他事，这个需求排不上

Why-5: 为什么排不上？
  → 自动化检查工具的优先级被低估
```

**🎯 根因**：缺少资产提交时的自动化面数检查，问题积累到最后才暴露

### 【影响范围】

| 维度 | 影响 |
|------|------|
| 工期 | 集中返工 10 天 |
| 成本 | 内部人力约 20 人天 |
| 质量 | 减面后部分角色细节损失 |

### 【解决方案】
1. 优先减面：裙摆背面、鞋底、不可见装饰区域
2. 建模组长逐个审核减面方案
3. TA 提供自动化减面脚本辅助

### 【预防措施】
1. ✅ 搭建 Pre-commit 面数检查 CI，超标自动拒绝
2. ✅ 每 Sprint 中期做一次面数抽检
3. ✅ 在任务卡片上标注面数红线

### 【经验金句】
> 💣 **"自动化检查永远比人工 Review 靠谱。问题越早发现，修复成本越低。"**
