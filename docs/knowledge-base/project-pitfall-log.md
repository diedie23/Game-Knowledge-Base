## 🚨 案例 1：外包翻车 — 首批角色交付全部打回

> **[S 级]** 外包管理 · 首次合作 CP · 量产启动阶段

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 涉及资产 | 5 个角色 |
| 延期天数 | +18 天 |
| 额外成本 | ¥3.5 万 |
| 打回率 | 80% |

签约 CP-Alpha（B 级供应商），下发 5 个角色建模需求。**仅提供参考图 + 口头说明**，无风格指南文档。CP 交付后主美评审：风格偏差大、面数超标、UV 不规范 — 4 个全部打回。双方扯皮 2 周，整体延期 18 天。

### 🎭 现场还原

- 💬 **APM**："签了 CP-Alpha，下发 5 个角色建模需求，提供参考图和简要口头说明。"
- 💬 **外包 CP**："按照你们给的参考图做的呀？"
- 💬 **主美**："风格偏差大，面数超标，UV 展开不规范，4 个全部打回！"
- 💬 **APM**：双方扯皮 2 周，整体延期 18 天……

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么 CP 交付风格偏差大？ | CP 只拿到参考图 + 口头说明，靠经验推测 |
| W2 | 为什么只给了参考图？ | APM 和主美未准备《风格指南》，认为参考图足够 |
| W3 | 为什么启动时没有风格指南？ | 外包启动流程中无"风格指南交付"强制节点 |
| W4 | 为什么没有首件样品提前暴露偏差？ | 为赶进度 5 个同时批量下发，无标杆件 |
| **W5** | **为什么无中期 Review？** | **APM 排期未预留 Review 时间 — 根因** |

### 🚀 破局思路

1. **外包启动前必须提供《风格指南》**（≥10 页）：含色板、材质标杆、UV 标准
2. **首件样品机制**：CP 先做 1 个标杆件通过后再批量
3. **每周 2 次在线 Review**：问题及时纠正
4. **合同增加首件不通过可解约条款**
5. **主美亲自做标杆角色**作为 CP 参照标准

### 🎬 制作人决策点

| 决策时机 | 制作人应做 |
|:---|:---|
| 外包启动评审 | 确认《风格指南》已交付，否则阻断启动 |
| 首件 Review | 亲自参与首件确认，把控品质方向 |
| 扯皮超 3 天 | 介入仲裁，果断决策：返工/验收/换 CP |

> 🔗 **深入阅读**：外包管理细节 → `cp-outsource.html` · 验收 Checklist → `cp-management.html`

## 🚨 案例 2：排期爆雷 — 场景组集体加班 3 周

> **[A 级]** 排期管理 · 方案未锁定就开始量产

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 涉及工种 | 场景组 6 人 |
| 加班时长 | 连续 3 周 |
| 人员流失 | 2 人提出调岗 |
| 返工比例 | 推倒重做 40% |

策划下发主城白盒方案 v1 → 场景组埋头做 2 周 → 策划改方案 → 推倒重做 40% → 制作人 3 周后才看到成品又要改动线 → 连续加班 3 周，2 人提出调岗。

### 🎭 现场还原

- 💬 **策划**："主城方案 v1 完成，开始白盒吧。"
- 💬 **场景组**：（埋头做了 2 周白盒）
- 💬 **策划**："方案改了，部分区域重新布局。"
- 💬 **场景组**：😱 推倒重做 40%...
- 💬 **制作人**：（3 周后才看）"入口区域动线不对，需要改。"
- 💬 **场景组**：💀 连续加班 3 周……

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么场景组推倒 40% 重做？ | 策划方案在白盒中途频繁变更 |
| W2 | 为什么策划能随意变更已开工方案？ | 方案未经制作人确认就下发，无 Gate Review |
| W3 | 为什么没有白盒三方确认？ | 流程缺少「制作人看白盒」强制节点 |
| W4 | 为什么大场景不做模块化分批确认？ | APM 没有拆分确认机制，全场景一次性下发 |
| **W5** | **为什么排期无缓冲应对变更？** | **APM 排期过于乐观，未预留 15~20% buffer — 根因** |

### 🚀 破局思路

1. **白盒三方确认**：制作人 + 策划 + 主美签字后才进美化
2. **方案变更走正式流程**：评估影响面后再执行
3. **大型场景模块化拆分**：按区域分批确认
4. **排期预留 15~20% 缓冲 buffer**
5. **「设计评审 → 冻结 → 生产」三阶段制度**

### 🎬 制作人决策点

| 决策时机 | 制作人应做 |
|:---|:---|
| 方案下发前 | **必须亲自签字**确认方案终稿，不能"最后再看" |
| 变更申请到达时 | 评估变更对工期/士气的影响，非致命变更 defer 到下一 Sprint |
| 团队加班超 1 周 | 主动介入调配资源或砍需求，防止人员流失 |

> 🔗 **深入阅读**：排期管理细节 → `art-scheduling.md`

## 🚨 案例 3：资产返工 — 角色面数超标被 QA 批量打回

> **[A 级]** 质量管控 · 制作过程中缺乏自动化检查

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 涉及资产 | 8 个角色 |
| 超标范围 | 18k~22k（红线 15k） |
| 返工耗时 | 5 天 |

建模师追求极致细节，8 个角色面数从红线 15k 膨胀到 18k~22k。TA 脚本最终走查才检查出来 → 全部打回减面。

### 🎭 现场还原

- 💬 **建模师**：（精雕细琢每个角色，追求极致细节）
- 💬 **TA 脚本**：⚠️ 面数检查 → 8 个角色超标
- 💬 **角色组长**："为什么中间没检查？"
- 💬 **TA**："自动化检查需求一直排不上优先级……"

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么 8 个角色面数超标？ | 建模师追求细节逐渐加面，无实时提醒 |
| W2 | 为什么制作过程中无人发现？ | 面数检查只在最终走查做，中间无卡点 |
| W3 | 为什么没有 Pre-commit 自动检查？ | TA 自动化工具优先级被低估 |
| **W4** | **为什么自动化优先级被低估？** | **APM 未意识到过程卡点比最终检查成本更低 — 根因** |

### 🚀 破局思路

1. **Pre-commit CI 面数检查**：超标自动拒绝提交
2. **每 Sprint 中期 TA 抽检**：不等到最终走查
3. **任务卡片标注面数红线**：建模师心中有数
4. **TA 提供自动化减面脚本**
5. **组长审核减面方案**：优先减不可见区域

### 🎬 制作人决策点

| 决策时机 | 制作人应做 |
|:---|:---|
| 工具优先级排期 | 将 CI 自动化卡点视为**基础设施投资**，优先级拉高 |
| 质量与效率权衡 | 明确"过程卡点"ROI > "最终走查返工成本" |

> 🔗 **深入阅读**：验收清单 → `art-vs-qa-checklist.md` · 提交审核流程 → `asset-submit-review.md`

## 🚨 案例 4：项目延期 — 量产排期连环爆雷

> **[S 级]** 排期管理 · 复杂度低估 + 需求膨胀 + 人力被挪用

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 涉及资产 | 12→14 个角色 |
| 延期 | +3 周 |
| 超支 | ¥12 万（追加外包） |
| 核心偏差 | 人天估算偏差 +68% |

Sprint 8 需在 4 周内完成 12 个 S 级角色。APM 沿用 B 级角色历史数据（2.5 天/个）评估，S 级实际需 4~5 天/个。中途策划口头新增 2 个角色 + 关键美术被借调 → 全面延期 3 周，追加外包 ¥12 万。

### 📅 事件时间线

| 时间 | 事件 | 信号 |
|:---|:---|:---|
| Week 1 Day 1-3 | 前 3 个 B 级角色按时完成 | 🟢 虚假安全感 |
| Week 1 Day 4-5 | S 级角色制作周期翻倍 | 🟡 进度开始滞后 |
| Week 2 Day 1 | 关键骨干被隔壁项目借调 2 天 | 🔴 产能下降 25% |
| Week 2 Day 3 | 策划微信口头新增 2 个角色 | 🔴 任务膨胀 +16.7% |
| Week 3 | 全面延期确认 | 💀 Beta 推迟 3 周 |

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么延期 3 周？ | S 级工时超评估 + 中途新增 + 人力借调 |
| W2 | 为什么工时超评估？ | 沿用 B 级数据，未按 S 级差异化评估 |
| W3 | 为什么没有按复杂度分级？ | 无标准化人天模型，排期依赖"拍脑袋" |
| W4 | 为什么新增需求没被拦截？ | 无变更管控流程，策划口头加需求 APM 碍于情面 |
| **W5** | **为什么关键人力被借调无预警？** | **项目间人力协调缺失 + APM 缺乏向上管理 — 根因** |

### 🚀 破局思路

1. **按 S/A/B/C 四级建立标准人天评估表**，每季度校准
2. **需求变更 CR 审批流程**：变更需评估影响后制作人签字
3. **冲刺期核心成员"锁定"**：借调需 APM 评估同意
4. **每周五燃尽图**：偏差 >20% 自动黄色预警

### 🎬 制作人决策点

| 决策时机 | 制作人应做 |
|:---|:---|
| Sprint 启动 | 审核排期是否按复杂度分级，人天模型是否合理 |
| 进度偏差 >20% | 立即召集复盘，决策方案：砍需求 / 加资源 / 延期 |
| 人力借调请求 | **冲刺期一票否决**，守住核心产能底线 |
| 口头加需求 | 坚持"不走 CR 不排期"原则，不因人情松口 |

### 📊 改进措施落地跟踪

| 措施 | 状态 |
|:---|:---|
| S/A/B/C 人天评估表 | ✅ 已落地 |
| CR 审批流程 | ✅ 已落地 |
| 关键人力锁定 | 🔄 推进中 |
| 周进度预警 | ✅ 已落地 |

## 🚨 案例 5：跨部门冲突 — 策划改需求引发美术返工风暴

> **[A 级]** 跨部门协同 · 风格大改 = 整条管线重做

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 涉及资产 | 8 个角色 |
| 返工耗时 | 2 周 |
| 人员流失 | 1 人递离职 |
| 前期浪费 | 32 人天归零 |

Sprint 6 最后一天，策划微信群通知："8 个核心角色从写实风改为赛璐璐二次元风格"。已完成的原画、PSD 拆分、Mask、Spine 全部需从头返工。

### 📅 事件时间线

| 时间 | 事件 |
|:---|:---|
| Sprint 5-6 | 8 个角色按原风格完成全流程通过 QA |
| Sprint 6 Day 10 | 策划微信群通知"风格全改"，未走变更流程 |
| Sprint 7 Day 1-2 | 美术团队情绪爆发，1 人递离职 |
| Sprint 7 Day 3 | 紧急三方会议，折中方案：4 改 + 4 保留 |
| Sprint 7-8 | 返工 2 周完成折中方案 |

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么 8 个角色要返工？ | 策划在角色完成后突然变更美术风格 |
| W2 | 为什么量产中期能大改风格？ | 项目未定义"风格冻结"节点 |
| W3 | 为什么变更没走正式评审？ | CR 模板有但从未推荐执行，策划习惯口头变更 |
| W4 | 为什么 APM 没第一时间拦截？ | 缺乏"说不"的勇气 |
| **W5** | **为什么策划不理解变更影响？** | **跨部门对美术管线成本缺乏共识 — 根因** |

### 🚀 破局思路

1. **风格冻结节点**：预研期结束前锁定，量产期不允许风格级变更
2. **CR 审批流推荐执行**：不走流程的变更 APM 有权拒绝
3. **变更成本速查表**：分发给策划团队，让"改风格 = X 人天"可见

### 🎬 制作人决策点

| 决策时机 | 制作人应做 |
|:---|:---|
| 预研结束评审 | 签字确认风格基调，**量产后风格不可改** |
| 策划提出风格变更 | 要求提交完整影响评估，非制作人批准不执行 |
| 团队情绪爆发 | 立即介入安抚 + 做决策（折中 / 拒绝 / 资源补偿） |

> 💡 **制作人金句**："你不替美术说'不'，就没人替你的团队说'不'。APM 是美术团队的最后一道防线。"

> 🔗 **深入阅读**：跨部门沟通话术 → `cross-dept-communication-tips.md` · 需求对接流程 → `art-vs-planner-req.md`

## 🚨 案例 6：外包质量崩盘 — CP 交付 100% 废弃

> **[S 级]** 外包管理 · 三重风险叠加（首次合作 + C 级 CP + 工期紧）

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 废弃资产 | 20 个角色 |
| 损失成本 | ¥18 万 |
| 不合格率 | 100% |
| 补位加班 | 2 周 |

量产后期为赶 CB1 版本，将 20 个 B 级角色外包给新合作 CP-07（C 级评级）。合同约定 3 周交付但未设中期检查节点。交付结果：面数超标 200%、命名全不合规、无 Mask 通道、贴图尺寸非 2^N → **全部废弃**。

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么全部不合格？ | CP-07 完全不了解我方管线规范 |
| W2 | 为什么 CP 不了解规范？ | 发包时只给需求列表，未附《资产制作规范包》 |
| W3 | 为什么没有中期检查？ | 3 周周期无中间卡点，直到最终交付才看结果 |
| W4 | 为什么选 C 级 CP？ | A/B 级产能已满，时间压力被迫降级 |
| **W5** | **为什么没有备选方案？** | **供应商池仅 3 家，未提前储备 — 根因** |

### 🚀 破局思路

1. **标准化《外包规范包》**：每次发包必附
2. **外包 > 2 周必设 50% 检查点**
3. **供应商池扩充**：每季度评估 2-3 家新 CP，A/B 级扩充到 8 家+
4. **新 CP 首单限量 ≤ 3 个资产**：试水验证后再放量

### 🎬 制作人决策点

| 决策时机 | 制作人应做 |
|:---|:---|
| 选择 C 级 CP | **一票否决** — 三重风险叠加时必须降低单次发包量 |
| 供应商池审查 | 季度 Review 供应商池健康度，A/B 级不足 5 家需扩充 |
| 损失 > ¥10 万 | 启动事故复盘会，输出制度级改进项 |

> 💡 **制作人金句**："外包管理的核心不是'把活甩出去'，而是'把标准传出去'。CP 的天花板 = 你发包规范的下限。"

### 📊 改进措施落地跟踪

| 措施 | 状态 |
|:---|:---|
| 标准化发包包 | ✅ 已落地 |
| 中期 Check 机制 | ✅ 已落地 |
| 供应商池扩充 | 🔄 推进中 |
| 新 CP 首单限量 | ✅ 已落地 |

## 🚨 案例 7：需求变更 — 封版前植入新系统引发连锁崩塌

> **[S 级]** 需求变更 · Feature Freeze 缺失 · 连锁效应

### 📋 事件概要

| 维度 | 数据 |
|:---|:---|
| 新增页面 | 15 个 UI |
| 延期 | +2 周 |
| 加班工时 | 200+ 人时 |
| 新增 Bug | 12 个（赶工导致） |

Beta 封版倒计时 7 天，制作人宣布根据市场反馈新增完整公会系统（15 个页面 + 美术资产）。美术 + 程序 + QA 三线全面承压 → 赶工导致已完成模块引入 12 个新 Bug → 被迫延期 2 周，公会系统精简为 8 核心页面。

### 🔍 5Whys 根因分析

| Why | 问题 | 答案 |
|:---:|:---|:---|
| W1 | 为什么 Beta 延期 2 周？ | 封版前 7 天植入全新系统，工作量远超剩余时间 |
| W2 | 为什么允许封版前新增？ | 无严格 Feature Freeze 机制 |
| W3 | 为什么制作人直接拍板不评估？ | "老板意见"被当免评估通行证 |
| W4 | 为什么公会系统没在早期规划？ | 产品规划与技术排期脱节 |
| **W5** | **为什么赶工导致更多 Bug？** | **极限压力下跳过 Code Review 和美术自检 — 根因** |

### 🚀 破局思路

1. **Feature Freeze**：封版前 2 周只接 Bug Fix，不接新功能
2. **变更成本公示**：新增需求必须附三方签字的《影响评估报告》
3. **版本分流**：来不及的需求自动进入"下一版本待办"

| 赶工超 3 天 | 评估质量下滑风险，果断决策延期而非带病上线 |

## 🏗️ APM 踩坑预防 5 道防线体系

> 总结 7 大案例，APM 应在项目中搭建以下防线
