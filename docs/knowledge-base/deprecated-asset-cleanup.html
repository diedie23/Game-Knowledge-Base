📦 1. 废弃资产定义与分类

核心问题：项目进行到中后期，SVN/Perforce 仓库往往膨胀到数百 GB 甚至 TB 级别，其中 30%~50% 可能是废弃资产。它们占用存储空间、拖慢同步速度、增加误引用风险。但"不敢删"是普遍心态——因为怕删错。本文档解决的就是"什么能删、怎么删、删完去哪找"的问题。
📊 1.1 废弃资产五大类型
类型代号定义典型来源占比估算
废案资产DEPRECATED曾投入制作但方案被否决、需求被取消的完整资产预研期换方向、策划砍功能、主美否定风格稿15~25%
过期版本OUTDATED已有更新版本替代的旧版文件迭代升级后的 v01/v02、被 v05 替代的早期版本20~30%
测试/临时文件TEMP测试用途、临时调试、WIP 快照等非正式文件测试场景、压力测试资产、截图、录屏10~15%
冗余副本DUPLICATE同一资产的多处拷贝（不同目录/不同命名）跨工种拷贝、外包交付后未清理原始包5~10%
孤儿资产ORPHAN引擎中无任何引用、不被任何关卡/预制件/蓝图使用的资产功能删除后残留的贴图/材质/Mesh5~15%
🔍 1.2 如何判断一个资产是否为"废弃"

📋 废弃判断 Checklist（满足 2 项即可标记为废弃候选）

☐JIRA/TAPD 中对应的任务状态为"已取消"或"需求已撤回"
☐SVN/P4 中该文件超过 90 天无任何提交修改
☐引擎 Reference Viewer 中无任何有效引用（孤儿资产）
☐已存在更新版本（如 v05 存在时 v01~v04 为过期版本）
☐文件名包含 _test / _temp / _bak / _old 等标记
☐制作人/主美/APM 口头或书面确认该资产不再需要
⚠️ 安全底线：即使满足多项判断条件，APM 也不得直接删除，必须走归档流程（§4）。"归档 ≠ 删除"——归档后的资产仍可按需恢复。

♻️ 5. 通用资产复用库建设与调用指南

核心价值：废弃不等于无用。很多废案资产中的模型拓扑、UV 布局、绑定骨架、材质方案仍具有复用价值。建立通用复用库（Asset Reuse Library），可以将"沉没成本"转化为"复用收益"，直接提升 ADEI。
📊 5.1 复用库分类体系
分类内容来源复用场景
🧱 通用模型库标准人体 BaseMesh、通用道具、基础建筑模块废案资产中的高质量基础模型新角色/场景的制作起点
🎨 材质预设库PBR 材质球预设、Substance 参数模板项目积累的材质方案快速搭建新资产材质

🎬 动画片段库通用待机/行走/跑步/攻击/受击动画废案角色的动画资产新角色的动画预置
✨ 特效模板库通用爆炸/命中/buff 光圈等特效预制件项目积累的特效方案新特效的制作基础
📐 UV/拓扑模板库标准 UV 布局模板、拓扑流向参考通过验收的高质量资产保证新资产拓扑质量

📋 5.3 复用库调用规范
步骤操作注意事项
① 检索在 _CATALOG.json 中搜索关键词支持工种/类型/风格/标签多维搜索
② 预览查看缩略图 + 技术参数（面数/骨骼数/贴图尺寸）确认是否满足当前项目规格要求

④ 改造基于复用模板进行定制化修改修改后的资产按正常流程提交到项目仓库

⚠️ 复用库入库标准：不是所有资产都能进复用库。入库资产必须满足：
· 通用性高（不含项目特有风格元素）
· 质量达 A 级以上
· 拓扑/UV/命名符合规范
· 附带使用说明和参数表

🤖 7. 自动化巡检与报表

⚙️ 7.1 自动化巡检脚本（Python 示例）
"""
废弃资产自动巡检脚本
功能：扫描 SVN 仓库，标记超过 N 天未修改的文件为废弃候选
"""
import os, json, subprocess
from datetime import datetime, timedelta

STALE_DAYS = 90          # 超过 90 天未修改视为候选
TEMP_PATTERNS = ['_test', '_temp', '_bak', '_old', 'Thumbs.db', '.DS_Store']
SCAN_ROOT = '/svn/Art/'  # 扫描根路径

def get_svn_last_modified(file_path):
    """获取文件最后修改时间"""
    result = subprocess.run(
        ['svn', 'info', '--show-item', 'last-changed-date', file_path],
        capture_output=True, text=True
    )
    date_str = result.stdout.strip()[:10]
    return datetime.strptime(date_str, '%Y-%m-%d')

def scan_stale_assets(root_path):
    """扫描超期未修改的资产"""
    stale_assets = []
    cutoff_date = datetime.now() - timedelta(days=STALE_DAYS)

    for root, dirs, files in os.walk(root_path):
        for f in files:
            full_path = os.path.join(root, f)

            # 检查临时文件模式
            is_temp = any(p in f.lower() for p in TEMP_PATTERNS)

            # 检查最后修改时间
            last_mod = get_svn_last_modified(full_path)
            is_stale = last_mod < cutoff_date

            if is_temp or is_stale:
                stale_assets.append({
                    'path': full_path,
                    'last_modified': last_mod.isoformat(),
                    'days_stale': (datetime.now() - last_mod).days,
                    'reason': 'TEMP' if is_temp else 'STALE',
                    'size_mb': os.path.getsize(full_path) / 1024 / 1024
                })

    return stale_assets

# 执行扫描并输出报告
results = scan_stale_assets(SCAN_ROOT)
print(f"扫描完成：发现 {len(results)} 个废弃候选资产")
print(f"预计可释放：{sum(a['size_mb'] for a in results):.1f} MB")

# 导出 JSON 报告
with open('stale_asset_report.json', 'w') as f:
    json.dump(results, f, indent=2, ensure_ascii=False)

📌 8. 典型案例与 Do / Don't

🔴 案例 1：误删在用资产，导致版本打包失败 致命

🎬 场景：TA 在清理孤儿资产时，将一组"看起来没被引用"的贴图删除。实际上这些贴图被动态加载（代码中用字符串路径引用），Reference Viewer 无法检测到。删除后版本打包时部分角色变成白模。

💡 教训：
① 动态加载的资产必须在资产注册表中登记，清理前交叉检查
② 清理后必须全量 Build 验证，确认无红色引用
③ 高风险清理（如贴图、材质）必须在独立分支上操作，验证无误后再 merge

🟡 案例 2：废案资产 6 个月后被要求"复活" 中频

🎬 场景：预研期被否决的 A 方案角色设计，在量产期被制作人重新拿出来说"其实这个方向也不错，能不能恢复？"。结果发现资产已被 Obliterate，无法恢复。

💡 教训：
① 归档 ≠ 删除。废弃资产默认走归档流程，保留至少 12 个月
② Obliterate 只用于 TEMP/DUPLICATE 类型，且需三方审批
③ 归档元数据中标注"复用潜力"，HIGH 潜力资产保留期延长至 24 个月

8.1 资产清理 Do / Don't

✅ Do（正确示范）

• 废弃先归档，不直接删除

• 归档前公示 3 天，征求团队异议

• 每个归档资产写 _ARCHIVE_META.json

• 清理后全量 Build 验证引用完整性

• 高复用潜力的废案提交到复用库

• 定期出存储健康报表

❌ Don't（错误示范）

❌ "这个文件看着没人用，直接删" → 可能是动态加载

❌ Obliterate 不审批 → 删错了无法恢复

❌ 从来不清理 → 仓库 2TB，同步 3 小时

❌ 只清引擎不清 SVN → 源文件仍占空间

❌ 废案直接扔掉 → 浪费可复用的模型/拓扑

❌ 清理后不通知团队 → 有人同步完发现文件没了
