📑 3. 各阶段详细说明

① 策划输出需求文档 原有

📥 Input
• 玩法设计文档 • 功能原型/草图
• 参考竞品截图 • 系统数据结构
📤 Output
• 完整 UI 需求文档（含页面清单）
• 功能逻辑说明 • 数据字段定义
• 优先级与排期预期
责任人：策划 预计工期：1-2 天

② 需求评审 NEW 补充

📥 Input
• 策划需求文档 • 竞品分析材料
• 技术可行性预评估
📤 Output
• 评审纪要（通过/打回）
• 需求冻结确认签字
• 各方排期承诺
参与人：策划交互UILayout客户端 预计：0.5 天
🔴 铁律：需求评审通过后进入冻结状态，后续变更需走正式变更流程。未经评审的需求禁止进入交互设计。

③ 交互设计 原有

📥 Input
• 已冻结的需求文档 • 平台设计规范
• 交互组件库
📤 Output
• 交互线框图（Figma/Axure）
• 页面跳转逻辑图 • 异常状态说明
• 手势/操作定义表
责任人：交互设计师 预计：3-5 天

④ 交互评审 → ⑤ 交互图交付 原有

📥 评审输入
• 交互稿 • 需求文档（对照）
• 技术可行性评估
📤 交付输出
• 正式交互标注文档（含尺寸、间距、状态）
• 交互动效说明 • 同步至蓝湖/Figma
参与人：策划交互UI客户端 预计：0.5 天评审 + 0.5 天交付

⑥ UI 设计 原有

📥 Input
• 交互标注文档 • 视觉风格指南
• UI 组件库/Design Token
📤 Output
• 高保真 UI 设计稿（含各状态）
• 标注文件 • 图标素材
• 多分辨率适配方案
责任人：UI 设计师 预计：5-7 天

⑦ UI 评审 → ⑧ UI 图交付 原有

📥 评审输入
• UI 设计稿 • 交互稿（对照）
📤 交付输出
• 切图资产（PNG/SVG，偶数尺寸）
• 标注文件（蓝湖/Figma）• 9宫格标注
参与人：策划交互UILayout 预计：0.5 天评审 + 1 天切图

⑨ 资产合规检查 NEW 补充

📥 Input
• UI 交付的切图资产包
• 资产命名规范文档
• 技术约束（最大尺寸、格式、颜色空间）
📤 Output
• 合规检查报告（PASS/FAIL）
• 不合规项清单 • 自动化校验日志
责任人：UILayout 预计：自动化脚本 30 秒
🔴 质量门禁：检查项：文件命名 模块_功能_状态_尺寸、偶数尺寸、PNG-24/WebP、sRGB、9宫格标注完整。

⑩ Layout 制作（功能蓝图 + 切正式资产 + 正式蓝图）原有

📥 Input
• 交互标注文档 • 合规通过的切图
• 引擎 UI 框架规范
📤 Output
• 功能蓝图（白模版）• 正式资产替换
• 正式蓝图（含动画/绑定/自适应）
责任人：Layout 工程师 预计：3-5 天
💡 并行：Layout 功能蓝图（白模）可与 UI 设计并行（交互交付后即启动），客户端可基于白模并行开发。

⑪ VX 动效（预演 + 正式制作）原有

📥 Input
• 交互动效说明 • UI 设计稿
• 引擎动效能力清单
📤 Output
• 动效预演视频/GIF
• 正式动效资产 • 动效参数文档
责任人：VX 动效设计师 预计：2-4 天

⑫ 动效评审 NEW 补充

📥 Input
• VX 动效预演视频 • 动效参数文档
• 性能预算表
📤 Output
• 评审纪要（通过/打回）
• 性能评估（DrawCall/帧率）
参与人：交互UIVX客户端 预计：0.5 天
⚠️ 动效是 UI 的"灵魂"也是性能杀手。评审必须同时关注表现效果和性能开销。

⑬ 客户端集成 → ⑭ 界面走查 → ⑮ 交付完成 原有

📥 集成输入
• 正式 Layout 蓝图 • 合规动效资产
• 功能逻辑+接口文档
📤 走查/交付输出
• 走查通过 → 资产归档 + 上线
• 问题清单 → 分流至对应责任人
集成：客户端 走查：策划交互UILayoutVXQA 预计：3-5 天集成 + 2-3 天走查

✅ 4. 关键评审节点与通过标准

评审节点参与人通过标准打回条件上限
②需求评审 NEW全员需求完整无歧义、技术可行、排期合理需求模糊、技术不可行2次
④交互评审策划+交互+UI+客户端逻辑完整、交互流畅、异常已处理遗漏分支、路径过深2次
⑦UI评审策划+交互+UI+Layout风格统一、标注准确、适配完整风格偏差、标注缺失2次
⑨资产合规 NEWUI+Layout命名100%合规、尺寸/格式通过、sRGB自动化检查FAIL不限
⑫动效评审 NEW交互+UI+VX+客户端动效与设计一致、性能在预算内DrawCall超预算2次
⑭界面走查全员像素级一致、功能正确、动效流畅P0/P1问题存在3次

🔴 铁律：超过最大迭代次数仍未通过 → 上升至项目经理介入。连续2次打回需根因分析。

🔀 5. 走查问题分流与闭环机制

问题类型典型表现分流至验收人SLA
🖌️ UI问题颜色偏差、图标错误、字体不一致、间距异常UI设计师主美/交互1天
📐 Layout问题蓝图错位、适配异常、层级错乱LayoutUI/交互0.5天
💻 开发问题功能逻辑错误、数据异常、接口未对接客户端策划/QA1-2天
🎨 交互问题流程不通、缺返回路径、反馈缺失交互策划1天
✨ 动效问题动效卡顿、方向不对、时长不合理VX交互/客户端1天
✅ 闭环流程：发现问题 → 填写修改清单（截图+描述+预期效果）→ 分流 → 修复 → 自测 → 复查 → 验收关闭 → 全部关闭后进入下一轮或交付。

❓ 7. 常见问题 FAQ

💬 Q1：策划中途改需求怎么办？
答：需求冻结后变更必须走正式流程：① 提交变更申请单 ② 各方评估影响 ③ 项目经理审批 ④ 更新需求文档标注变更。未审批的变更 = 不执行。
💬 Q2：小团队也要走这么多评审吗？
答：评审可以简化但不能跳过。小团队可以：合并④⑤为一个快速评审、合并⑦⑧为一个快速评审。但⑨资产合规（自动化30秒）和⑭界面走查绝不能省。
💬 Q3：Layout 白模和 UI 设计并行，如果 UI 改了怎么办？
答：白模只验证功能逻辑，不依赖最终视觉效果。UI 评审通过后 Layout 切正式资产时才替换。如果 UI 改动涉及布局结构变化，则需同步通知 Layout 调整白模。
💬 Q4：走查发现大量问题，是否需要重新走全流程？
答：不需要。走查问题按类型分流修复后进行复查，而非重走全流程。但如果问题是系统性的（如需求理解偏差），则需评估是否回退到对应阶段重做。
