## 1️⃣ 吸色与通道分配交互重构

### 问题回顾
v6.0 中，用户在画布上吸取颜色后，需要在右下角找到很小的「➕ 添加」按钮，再选择目标通道。操作路径长（吸色 → 滚动面板 → 选通道 → 点添加），反直觉。

### v6.1 优化方案

#### 🔹 悬浮快捷菜单 (Float Menu)
- **触发方式：** 在画布上点击吸取颜色后，**立即**在鼠标位置附近弹出悬浮菜单
- **菜单内容：**
  - 当前吸取颜色的色块 + HEX 值 + 匹配百分比
  - `[分配给 R]` `[分配给 G]` `[分配给 B]` `[分配给 A]`（带颜色标识点）
  - `[➕ 新建自定义通道]`
- **交互行为：** 点击任一按钮 → 自动切换到对应通道 → 添加颜色区域 → 菜单消失
- **关闭方式：** 点击菜单外任意区域自动关闭

#### 🔹 右侧面板"当前吸取颜色"展示区
- **位置：** 右侧参数面板最顶部（色相范围滑块上方）
- **展示内容：**
  - 40×40px 大色块 + HEX 代码（16px 大字）
  - HSV 三分量数值
  - 匹配百分比
  - 大按钮行：每个通道一个带颜色标识的分配按钮

### 使用流程
```
1. 在画布上点击目标颜色
2. 方式 A：在弹出的悬浮菜单中直接点击目标通道
3. 方式 B：在右侧面板顶部的展示区中点击大按钮分配
4. 颜色自动添加到指定通道 ✅
```

## 2️⃣ 智能分析主色调 — 消除"幽灵"反馈

### 问题回顾
v6.0 中点击「✨ 智能分析主色调」后，只在右下角弹出"发现4个主色域"的文字提示，但实际上没有任何颜色被分配到通道，用户看不到结果，需要手动逐个点击色块。

### v6.1 优化方案

#### 🔹 自动分配到基础通道
- 分析完成后，**自动**将检测到的主色调按占比排序，依次分配到 R → G → B → A 通道
- 避免重复添加（如果该色调已在某通道中，则跳过）

#### 🔹 结果展示面板
- 在「智能分析」按钮下方展开一个结果面板
- 展示提取出的色块（颜色 + 色相角度 + 占比百分比 + 目标通道）
- 提供 **[✅ 一键应用到基础通道 (R/G/B/A)]** 显式按钮
- 点击按钮后自动触发 Mask 生成

#### 🔹 画布视觉反馈
- 分析完成瞬间，在画布上叠加一层半透明色彩覆盖层
- 每个匹配区域以对应主色调的颜色闪烁 1.5 秒
- 通过 CSS animation (`flash-overlay`) 实现渐隐效果

### 使用流程
```
1. 加载图片 → 进入吸色步骤
2. 点击 [✨ 智能分析主色调]
3. 画布上立即闪烁高亮匹配区域（视觉确认）
4. 颜色自动分配到 R/G/B/A（Toast 提示）
5. 可选：点击 [✅ 一键应用到基础通道] 重新确认
6. 点击 [🔬 生成多通道 Mask] 完成
```

## 3️⃣ 图片裁剪引擎 — DPR 感知精准裁剪

### 问题回顾
v6.0 的裁剪在高 DPI 屏幕（如 4K 显示器、MacBook Retina）上出现坐标偏移，裁剪结果与选区不一致。根因：
1. `toDataURL() → new Image()` 链路引入二次编码损失
2. 未区分 CSS 像素与设备物理像素

### v6.1 修复方案

#### 🔹 坐标系明确化
```
坐标系说明：
  st.rx/ry/rw/rh → 始终为原图像素坐标（与 img.width/img.height 对应）
  不受 CSS zoom / devicePixelRatio / 容器缩放影响
  输出 canvas 尺寸 = 裁剪区域像素尺寸
```

#### 🔹 无损裁剪管线
```
原图 img
  ↓ drawImage(img, rx, ry, rw, rh, 0, 0, rw, rh)  ← 1:1 像素裁剪，无缩放
离屏 canvas (rw × rh)
  ↓ getImageData()  ← 直接读取像素数据
  ↓ putImageData()  ← 无损写入新 canvas
  ↓ toBlob('image/png')  ← PNG 无损编码
  ↓ createObjectURL()
最终 Image 对象  ← 零质量损失
```

#### 🔹 关键代码注释
```javascript
// canvas.width/height 设为目标像素尺寸（与 DPR 无关！）
// imageSmoothingEnabled = false 防止缩放导致像素模糊
// drawImage 9 参数形式：src 和 dst 尺寸相同 = 1:1 像素级裁剪
// 使用 toBlob → createObjectURL 替代 toDataURL 避免二次编码损失
```

## 4️⃣ 自定义通道命名

### 问题回顾
v6.0 中添加自定义通道后名称固定为 `Mask 1`、`Mask 2`，导出时文件名无法区分用途。

### v6.1 优化方案

#### 🔹 可编辑名称输入框
- 自定义通道卡片头部，名称区域改为可编辑的 `<input>` 文本框
- 默认值：`Mask 1`（递增）
- 支持自由修改为有语义的名称，如 `Emission`、`Metallic`、`Roughness`
- placeholder 提示："输入自定义名称(如Emission)"
- 修改后实时同步到：通道选择栏、参数面板、导出按钮、画笔通道选择器

#### 🔹 导出文件名联动
- 导出命名模板中 `{Channel}` 变量自动使用自定义名称
- 特殊字符（`\ / : * ? " < > |` 及空格）自动替换为下划线 `_`
- 示例：
  ```
  自定义名称: Emission
  模板: {Project}_{CharID}_Mask_{Channel}
  输出: MyGame_Hero01_Mask_Emission.png
  ```

#### 🔹 工程文件兼容
- `config.json` 中的 `channels` 数组新增 `isCustom` 字段
- v6.0 工程文件导入时自动兼容（无 `isCustom` 字段视为基础通道）
