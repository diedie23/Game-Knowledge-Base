📋 3. TAPD 发单机制与职责分工

UI管线涉及多角色协作，TAPD单据的创建和流转直接影响进度跟踪和复盘效率。推荐采用「策划创建父单 + APM拆解子单」的协作模式：

📝 策划 — 创建父需求单
时机：需求评审通过后
单据类型：TAPD需求（父级）
必填字段：
• 标题：[UI] 界面名称 — 功能简述
• 详细描述：需求文档链接、页面清单
• 优先级/迭代版本/截止日期
• 关联文档：Figma链接
处理人：设为对应策划本人（便于策划时刻跟踪该系统/功能的整体状态与进度，各环节子单进度会向上汇总。可抄送或@交互设计师提示启动设计）
📊 APM — 拆解子单 & 进度跟踪
时机：收到父需求单后（1个工作日内）
拆解子单清单：
• 交互设计 → 处理人：交互设计师
• UI设计 → 处理人：UI设计师
• Layout蓝图 → 处理人：Layout工程师
• VX动效 → 处理人：VX动效设计师
• 客户端集成 → 处理人：客户端程序
• 界面走查 → 处理人：策划/UI设计/Layout负责人
关键动作：设置子单依赖关系、里程碑、预警
⏱️ 发单时序与流转规则
环节阶段触发动作 / 里程碑TAPD 单据状态流转操作人
①→②需求文档定稿策划建父需求单，状态设为 已受理策划
②启动界面设计拆解子单（交互/UI/Layout/动效等），明确前置依赖与排期APM
③→④交互评审通过 & 交付交互子单 → 已完成，UI / Layout白模子单 → 进行中交互 / APM
⑥→⑨UI评审通过 & 资产合规UI子单 → 已完成，Layout切正式单 → 进行中UI / QA
⑩→⑫正式蓝图与VX资产就绪Layout蓝图子单 → 已完成，VX制作子单 → 已完成Layout / VX
⑭→⑰走查验收与客户端集成客户端集成单 → 已完成，父需求单验收 → 已完成客户端 / 策划
⚠️ 发单与流转注意事项：
• 父单由策划创建（确保需求来源可追溯），子单由APM拆解（确保排期合理、前置依赖正确）。
• 每个子单必须强关联父需求单，便于项目看板追踪系统整体研发进度。
• 评审类节点（③⑦⑭）通过 TAPD 评论/备注记录结论与调整点，不单独建单。
• 走查发现的问题统一通过 TAPD 缺陷单记录并关联父需求单。提报规范：必须包含【问题截图/录屏】、【期望表现】及【当前表现】，避免沟通拉扯造成返工。

❓ 4. 常见问题 FAQ

💬 Q1：策划中途改需求怎么办？
答：需求冻结后变更必须走正式流程：① 提交变更申请单 ② 各方评估影响 ③ 项目经理审批 ④ 更新需求文档标注变更。未审批的变更 = 不执行。
💬 Q2：小团队也要走这么多评审吗？
答：评审可以简化但不能跳过。小团队可以：合并③④为一个快速评审、合并⑦⑧为一个快速评审。但⑨资产合规（自动化30秒）和⑭界面走查绝不能省。
💬 Q3：Layout白模和UI设计并行，如果UI改了怎么办？
答：分两种情况——
① 仅视觉层变更（配色/字体/图标样式）：白模不受影响，Layout切正式资产时直接替换即可。
② 布局/结构性变更（控件增减、层级调整、页面布局大改）：这是并行模式下的高风险场景。此时白模功能蓝图可能无法使用，需要触发「偏离评估会」。
预防关键：UI设计进行至50%时必须与交互/Layout做一次中期对齐，提前暴露布局变更意图。
💬 Q4：走查发现大量问题，是否需要重新走全流程？
答：不需要。走查问题按类型分流修复后进行复查，而非重走全流程。但如果问题是系统性的（如需求理解偏差），则需评估是否回退到对应阶段重做。
💬 Q5：拿到UI视觉稿后，发现跟交互稿出入很大，程序说功能蓝图没法用，怎么办？ NEW
答：这是并行管线中最典型的返工风险。处理步骤：
第一步 · 紧急评估（≤半天）：交互+UI+Layout+客户端四方快速对齐，量化偏离范围。
第二步 · 责任判定：
　• 如果UI未经沟通擅自改变布局 → UI承担返工责任
　• 如果交互稿粒度不足，UI合理补充 → 交互确认变更，返工工期由项目承担
第三步 · Layout返工（+2~3天）：Layout基于UI终稿重新搭建白模蓝图。
预防机制：UI设计50%时做一次中期设计对齐，30分钟快速同步布局意图，可避免80%以上的此类返工。
💬 Q6：测试发现界面 Bug，TAPD 缺陷单应该挂给谁？ NEW
答：为了提高问题处理效率，请遵循以下分流原则：
　• 界面表现相关（含 Layout 还原/布局 及 视觉切图/资源错误）：统一挂单给 v_zhyszhang（张荧）。由张荧统一排查后，分配给对应的 Layout 制作人或 UI 设计师跟进处理。
　• 业务逻辑/数据异常问题：挂单给对应的客户端开发。
　• 界面表现相关但不确定归属的问题：优先挂单给 v_zhyszhang（张荧）协助定位排查。
其他明确不属于界面的常规问题，按已有逻辑挂给对应接口人即可。
💬 Q7：Layout 新手问题反复出现，如何保障蓝图还原质量？ NEW
答：Layout 新手容易出现同类问题反复、走查修改率高、制作周期拉长等情况。建议从制作前规范约束、制作中过程管控、制作后走查复盘三个阶段建立质量保障机制：

一、制作前 — 规范先行 & 任务拆解
　• 建立 Layout 制作 Checklist：将高频问题整理成标准检查清单（如：锚点设置、适配规则、命名规范、图集引用、Drawcall 控制等），新手在提交前必须逐项自查。
　• 任务分级指派：按界面复杂度分为 S/A/B/C 级，新手优先分配 B/C 级（简单弹窗、通用列表等），积累经验后再逐步承接 A/S 级（复杂主界面、多状态页签）。
　• 制作前对齐会（5~10分钟）：新手接单后，由 Layout 接口人快速过一遍 UI 稿要点，提前指出易踩坑的地方（特殊适配、动态列表、多语言文本溢出等），避免闷头做完才发现方向错。

二、制作中 — 过程管控 & 阶段性审查
　• 关键节点中期检查：对于 A 级及以上的界面，新手完成 50% 时必须提交一次中期截图/录屏给 Layout 接口人审查，及时纠偏，避免最终走查时大面积返工。
　• "同类问题不过三"机制：Layout 接口人对走查中发现的问题进行分类统计，若同一类型问题在同一人身上出现 ≥3 次，需要启动一次专项 1v1 辅导，找到根因（是规范不清楚？工具不熟练？还是理解偏差？）并针对性解决。
　• 结对制作（Pair Work）：对高复杂度界面，安排新手与资深 Layout 结对制作，新手主做、老手旁审，现场答疑，加速经验传递。

三、制作后 — 走查标准化 & 问题沉淀
　• 分层走查机制：新手产出的蓝图必须先经过Layout 接口人内审（内部走查），通过后再进入正式界面走查环节（⑭），减少无效的外部走查轮次。
　• 走查问题归档 & 典型案例库：每次走查结束后，将典型问题（截图+原因+修复方案）沉淀到团队共享文档中，形成可复用的「避坑案例库」，新人入职时必读。
　• 阶段性复盘：每 2~4 周对走查修改率做一次数据统计（走查通过率 / 单次修改量 / 返工次数），跟踪每位新手的成长曲线，用数据驱动培养计划调整。

💡 核心理念：不要等走查阶段才发现问题——把质量关口前移。通过 Checklist 自查 + 中期检查 + 内审三道防线，可以大幅降低新手的返工率，同时加速其成长速度。
