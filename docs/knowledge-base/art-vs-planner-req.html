🔗 1.4 并行流转与依赖管理（v3.0 新增）

复杂需求涉及多工种串联或并行时，需明确依赖关系和触发机制：

依赖类型定义处理策略系统支持
强依赖（FS）上游完成后才能开始下游上游验收通过自动触发下游启动需求链自动编排
弱依赖（SS）可同时开始但需阶段对齐设置同步检查点，定期对齐关联需求状态联动通知
外部依赖依赖其他部门/外部资源标记阻塞，挂起 SLA 计时每日自动催促依赖方
资源依赖需等待特定人员/工具就位APM 提前协调资源窗口资源日历冲突检测
⚠️ 依赖管理红线：任何需求的强依赖链长度不得超过 4 级。超过 4 级必须拆分为独立可交付子项目。

📊 2.5 需求质量评分卡

需求单提交后系统自动评分，≥ 70 分方可进入评审池，低于 70 分退回修改：

评分维度权重满分标准扣分规则
字段完整性25%所有必填字段均已填写每缺1个必填字段 -5 分
描述清晰度20%需求描述含背景/目标/约束三要素缺一要素 -7 分
参考图质量25%≥3张 + 有标注 + 风格一致按 2.4 参考图规格逐项评
验收标准15%含 ≥3 条可量化验收条件每少1条 -5 分
技术约束15%明确面数/尺寸/帧数等技术限制未填写技术约束 -10 分
💡 评分示例：一份需求单所有字段填满(25) + 三要素齐全(20) + 3张参考图有标注(22) + 4条验收标准(15) + 明确技术约束(15) = 97分 ✅ 通过。系统同时输出改进建议帮助策划持续提升。

满足以下全部条件的需求可跳过评审会，由 APM 直接审批排期：

条件标准判定人

质量评分≥ 85 分系统自动

🤯 8. 常见冲突场景与解法

🔴 案例 1：需求描述不清 高频

🎬 场景还原：策划说"做一个好看的角色"、"参考某游戏的感觉"，美术无从下手。

🔍 根因分析（5 Whys）：

① 表面：需求描述模糊、参考图不足（策划执行问题）
② 中层：策划缺乏美术语言，不知道怎么描述（能力缺失）
③ 深层：缺少需求单填写指南和培训（流程缺陷）
④ 系统层：没有质量门禁阻止低质量需求进入评审（机制缺失）

💡 系统性解决方案：

1. 质量评分卡门禁（v2.0 新增）：< 70 分自动退回
2. APM 主动出击，要求补充参考图
3. 组织 15min 面对面沟通，产出美术理解确认书
4. 建立 需求单填写培训机制，新入职策划必须学习后才能提交需求
5. 维护 优秀需求单示例库，策划可直接参考填写

📊 预期效果：退回率从 40% 降至 15% 以下；首次通过率从 30% 提升至 65%。

🔴 案例 2：策划频繁变更 高频

🎬 场景还原：原画做到一半改设定，建模快完成改造型。

🔍 根因分析（5 Whys）：

① 表面：原画做到一半改设定（结果）
② 中层：策划设计未收敛就下需求 / 上级反复审核（流程前置不足）
③ 深层：缺乏变更成本可视化机制，变更"零成本感"（机制缺失）
④ 系统层：绩效不关联变更率，无负反馈（激励失当）

💡 系统性解决方案：

1. 所有变更走 TAPD 工单，留痕追踪
2. 量化变更成本，每次变更标注影响人天 × 代价系数
3. 建立 变更成本可视化看板，让决策者在知道代价的前提下做决定
4. 定期汇报变更次数和代价，形成压力
5. 将变更率纳入策划侧 过程质量考核 参考指标

📊 预期效果：月变更率从 25% 降至 10% 以下；变更引发的返工人天减少 60%。

🔴 案例 3：优先级争议 高频

🎬 场景还原：所有需求都标 P0，排期陷入混乱。

🔍 根因分析（5 Whys）：

① 表面：所有需求都标 P0（滥用）
② 中层：策划之间没有统一的优先级标准（共识缺失）
③ 深层：缺乏统一的优先级仲裁机制 + 数量上限（规则缺失）
④ 系统层：P0 无上限约束，"通货膨胀"无代价（系统缺陷）

💡 系统性解决方案：

1. P0 全项目上限 3 个，超标自动预警制作人仲裁
2. 建立 优先级矩阵（紧急×重要×里程碑关联度）
3. 每周制作人会上 统一排序 + 公示
4. 在需求评审会上强制执行 P0 上限
5. P1 不超过总需求量 30%，超标触发复审

📊 预期效果：P0 需求数稳定控制在 3 个以内；排期冲突减少 70%。

🟡 案例 4：验收标准不一致 中频

🎬 场景还原：美术按需求单做完，策划验收时说"和我想的不一样"，但需求单上没写清楚。

🔍 根因分析：验收标准模糊/主观化；缺少"美术理解确认"环节；未在制作中期做 Check-in。

💡 解决方案：

1. 需求单必须包含 ≥3 条可量化验收标准（如"配色暖色系为主""面数 ≤ 15K"）
2. 制作前产出 美术理解确认书，策划签字认可
3. 制作中期（50% 进度）做 WIP Review，策划确认方向正确
4. 验收时对照 Checklist 逐条打分，非主观感受

🟡 案例 5：跨工种需求协调困难 中频

🎬 场景还原：一个角色需要原画→建模→动画→特效四个工种串联，但各工种信息不同步，前一环节的修改没传递到后一环节。

🔍 根因分析：信息在工种间传递时衰减；缺少统一的需求单关联机制；各工种只看自己那张需求单。

💡 解决方案：

1. 跨工种需求必须建立 需求链（Parent-Child 关联）
2. APM 指定一个 需求链 Owner 负责全链路协调
3. 上游工种完成后 自动触发下游启动通知
4. 任何一环变更 自动通知全链路相关人

🟡 案例 6：需求积压导致排期崩溃 中频

🎬 场景还原：Sprint 中间突然涌入大量 P0/P1 需求，现有排期被打乱，美术不知道先做哪个。

🔍 根因分析：缺乏产能容量模型；没有需求准入上限；优先级仲裁机制执行不力。

💡 系统性解决方案：

1. 建立 产能容量模型，过载自动预警并暂停接收新需求
2. 设置 Sprint 中期锁定线：Sprint 过半后仅接 P0
3. 制作人每周一 统一优先级排序，公示给全员
4. 需求缓冲区：预留 20% 产能应对突发需求
5. 定期复盘 需求到达率和团队消化率的匹配度

📊 预期效果：排期稳定性从 60% 提升至 85%；Sprint 完成率从 70% 提升至 90%。

🟡 案例 7：美术做完策划不验收 中频

🎬 场景还原：美术按期交付了成果，但策划迟迟不验收，需求卡在"验收中"状态长达一周甚至更久。

🔍 根因分析：策划忙于其他工作忽略验收；验收 SLA 执行不严；缺少激励/惩罚机制。

💡 系统性解决方案：

1. 验收 SLA 硬约束：超 3d 未反馈自动视为通过（已实施）
2. 验收任务纳入策划 每日待办 Top3 推送
3. 每周汇报 策划验收及时率排行榜
4. WIP CP3 检查时策划已参与，正式验收时无新信息 → 应快速决策

📊 预期效果：验收阶段平均耗时从 4d 降至 1.5d。

💡 9.4 数据驱动决策案例（v3.0 新增）

📊 案例 A：数据发现"某策划退回率 60%"

📊 案例 B：数据发现"角色建模工期持续偏高"

根因定位：预估模型中未包含"高模→低模拓扑"的工时；TA 审核环节平均等待 1.5d。

改进动作：① 更新工期估算基准（+30% 拓扑工时）② TA 审核从串行改为并行（建模 80% 时提前介入）。

改进结果：实际/预估比降至 1.1 倍；交付准时率从 40% 提升至 85%。
