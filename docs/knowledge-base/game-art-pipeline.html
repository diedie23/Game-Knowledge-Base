🔗 1. 全链路 Pipeline 总览

从策划需求到引擎最终表现的完整美术生产线，共 8 个核心阶段：

①需求
策划文档
→
②原画
概念设计
→
③拆分
PSD分层
→
④工具链
Mask/Spine/切图
→
⑤QA验证
自动化+人工
→
⑥引擎导入
资产配置
→
⑦材质/表现
Shader接入
→
⑧发布
构建包验证
核心原则：每个阶段都有明确的Input（上游交付物）和Output（本阶段交付物）。上游的 Output = 下游的 Input。契约不满足 → 不允许进入下一阶段。

🔴 铁律：Pipeline 的 8 个阶段可以合并但不能跳过。尤其是 ⑤QA 验证，自动化脚本跑一次只要 30 秒，但能避免 90% 的返工。跳过 QA = 引擎翻车。

📑 2. 各阶段 Input/Output 契约

① 需求阶段 — 策划 → 美术

📥 Input
• 策划文档（功能描述）
• 角色/场景/UI 需求列表
• 参考图/情绪板
• 数量和优先级排期
📤 Output
• 美术制作 Brief
• 风格定义文档
• 技术规范摘要
• 里程碑排期表

② 原画阶段 — 概念设计

📥 Input
• 美术 Brief
• 项目风格指南
• 色板 .aco
• 体型/头身比参考
📤 Output
• 正面/侧面/背面三视图
• 色板标注
• 表情 Sheet（如需）
• 主美签字确认

③ PSD 拆分阶段 — 部件分层

📥 Input
• 已确认的原画
• 部件拆分清单
• Z-Order 层级表
• Padding 规范值
📤 Output
• 分层 PSD（每部件一个图层组）
• 素体完整安全区
• Z-Order 标注文件
• 自检 Checklist ✅

④ 工具链阶段 — 自动化处理

📥 Input
• 合规的 PSD / PNG
• Mask 通道定义
• 命名模板配置
• 拓扑参数（Spine 用）
📤 Output
• Mask 遮罩图 PNG
• Spine 部件切片
• UI 切图（偶数尺寸）
• config.json 工程文件
• 命名合规的文件包

⑤ QA 验证阶段 — 质量门禁

📥 Input
• 工具链输出的资产包
• QA 阈值配置
• 工程验收 Checklist
📤 Output
• QA 报告（PASS/FAIL）
• 自动化校验日志
• 不合格项标注
• 视觉审核意见（如需）

⑥ 引擎导入阶段 — 资产配置

📥 Input
• QA 通过的资产包
• 引擎导入设置表
（sRGB/Filter/压缩/Mipmap）
📤 Output
• 引擎内资产（.uasset / .asset）
• 导入验证截图
• 纹理压缩后内存报告

⑦ 材质/Shader 接入阶段

📥 Input
• 引擎内资产
• 材质参数表
• 换色配色 JSON
• 动画控制器
📤 Output
• 材质实例
• 换色效果 3 套预览
• 动画播放验证
• 性能 Profiling 报告

⑧ 发布阶段 — 构建包验证

📥 Input
• 材质/动画 OK 的资产
• 多端真机列表
• 性能预算表
📤 Output
• 真机走查报告
• 包体大小报告
• 最终 SVN/Git 提交
• Release Note

⚙️ Pipeline 阶段流转自检

☐需求冻结：策划 Brief 已 Sign-off，不再变更
☐原画确认：三视图 + 色板已经主美签字
☐PSD 合规：图层分组正确、Z-Order 标注完整
☐工具链完成：Mask/Spine/切图全部输出、命名合规
☐QA 通过：自动化 + 人工验收全部 PASS
☐引擎验证：导入无报错、sRGB/Filter 设置正确
☐材质达标：换色效果/动画播放/性能 Profiling 均通过
☐发布就绪：真机走查通过、包体大小达标

✅❌ 3. 图文红黑榜

3.1 跳过 QA 门禁 → 引擎翻车

❌ Don't — 原画直接扔给程序"你看着导"

[此处插入图片：美术把未经拆分、未经QA的 PSD 直接发给程序。程序导入后 Mask 通道全错、命名混乱、尺寸非 POT，引擎中表现完全不对]

后果：跳过③④⑤三个阶段，程序只能"猜"美术意图，猜错了双方反复沟通——一个资产的集成周期从 1 天膨胀到 1 周。

✅ Do — 严格走完 8 阶段 Pipeline

[此处插入图片：Pipeline 看板截图，每个资产清晰标注当前阶段，每个阶段有明确的 Input/Output 和验收签字]

方法：使用项目管理工具（TAPD/Jira）创建 Pipeline 看板，每个资产卡片必须按序流转，上游签字后才能进入下游。

3.2 Input/Output 不定义 → 反复返工

❌ Don't — 口头传达需求，无书面契约

[此处插入图片：微信语音截图"帮我画个角色"，没有 Brief、没有规范、没有尺寸要求，结果返工 5 次]

后果：每个人理解不同，交付物不符合下游预期。原画觉得完成了，但 PSD 分层不对；PSD 分层好了，但命名格式不对。每个衔接点都可能返工。

✅ Do — 每个阶段有书面 Input/Output 契约

[此处插入图片：规范的 Pipeline 契约文档，清晰列出每阶段的 Input 清单、Output 清单、验收标准]

方法：本文档第 2 章的 8 个阶段 I/O 表即为标准契约。打印贴在工位上，每次交接核对。缺少任何一项 Input = 不开工。缺少任何一项 Output = 不交接。

🎮 4. 各游戏类型 Pipeline 差异

游戏类型核心差异重点阶段特殊工具
二次元手游角色立绘精度高、换装/换色需求大②原画（精修）、④Mask生成、⑦换色Shader自动 Mask 生成器、换色预览工具
轻度 2D 休闲角色数量巨大但单个复杂度低、UI 占比大③批量拆分、④UI切图、⑤自动化QA批量切图脚本、Atlas 自动打包
重度 3D 游戏PBR 材质流程、LOD 多级、光照烘焙②高模→低模、④法线烘焙、⑥LOD配置Substance、Marmoset、引擎烘焙
Spine 动画向骨骼绑定、网格变形、Atlas 打包③部件拆分、④Spine工具链、⑤Atlas QASpine 拆分工具、Atlas 验证器
⚠️ 提示：不同类型项目共用同一 Pipeline 框架（8 阶段），但每个阶段的具体交付物和验收标准可能不同。项目启动时需根据类型定制各阶段的 Checklist。

❓ 5. 常见问题 FAQ

💬 Q1：Pipeline 太重了，小团队也要走 8 个阶段吗？

答：阶段可以合并但不能跳过。小团队可以：① 将②③合并（原画+拆分同一人完成） ② 将⑥⑦合并（导入+材质同步配置） ③ 但⑤QA 验证绝不能跳——自动化脚本跑一次只要 30 秒，但能避免 90% 的返工。

💬 Q2：如何加速 Pipeline 周转效率？

三大加速器：

① 工具自动化：Mask 自动生成（替代手动逐通道涂画）、批量命名校验、一键导出工程包。

② 模板化：预设好 PSD 模板（含安全区辅助线）、命名模板、引擎导入 Preset。

③ 并行化：原画确认后，拆分和 Mask 制作可并行；不同角色的 Pipeline 可并行。关键路径上不要有串行瓶颈。

💬 Q3：不同阶段由不同人负责，如何保证一致性？

答：靠契约+工具而非口头约定：

① 每个阶段的 I/O 契约写成文档，全员可查。

② 工程规范通过自动化脚本强制执行（CI/CD 拦截不合规提交）。

③ 每周 Pipeline Review——TA Lead 抽检各阶段交付物，发现流程偏差及时修正。
