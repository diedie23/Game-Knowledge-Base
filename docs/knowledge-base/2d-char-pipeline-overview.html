2.2 设计稿交付标准
#交付物格式说明
1正面全身设计稿PSD/PSB (分层)必须保留完整图层结构
2表情 SheetPNG / PSD至少 4 种表情（默认/喜/怒/惊）
3色彩方案PNG + 色值标注默认配色 + 2~3 套变体配色
4动作参考GIF/视频/手绘稿关键动作的运动参考（可选但推荐）
5三视图/转面图PNG3D 感强的角色必须提供侧面参考
⚠️ 常见返工原因 #1：设计稿画好后才发现"这个发型不适合做骨骼动画"或"这个配饰结构太复杂拆不了"。解决方案：原画阶段就让动效师参与评审，在线稿阶段识别骨骼可行性问题。

3.2 Spine vs Live2D 选型决策
维度SpineLive2D
面部表情⭐⭐⭐ 通过 Skin 切换实现⭐⭐⭐⭐⭐ 参数驱动，极其自然
身体动态⭐⭐⭐⭐⭐ 骨骼驱动，自由度高⭐⭐⭐ 网格变形为主，大幅度运动受限
战斗动画⭐⭐⭐⭐⭐ 原生支持多动画混合⭐⭐ 不擅长复杂动作
性能（同规格）⭐⭐⭐⭐ CPU 蒙皮，轻量⭐⭐⭐ SDK 较重，内存占用高
换装支持⭐⭐⭐⭐ Skin 系统成熟⭐⭐ 需要通过 Part 切换
学习成本⭐⭐⭐ 中等⭐⭐ 较高（参数多）
推荐场景战斗+换装+多角色立绘展示+VTuber+AVG对话

5.2 关键协作节点

✅ 正确协作 — 原画阶段就拉动效师评审

在线稿阶段让动效师看一眼结构可行性。例如："这个飘带能不能做骨骼？""这个发型旋转会不会露出空白？"提前发现问题，避免完稿后才返工。

❌ 错误协作 — 原画完稿后甩图给动效

设计稿画完了才交给动效师，结果发现："手臂和身体重叠区域没预留""这个配饰结构做不了旋转"。原画返工 3 天。

7.2 版本管理策略
场景规范说明
日常提交每完成一个阶段提交一次（P3完成/P5完成/P7完成）Commit Message 格式：feat(char): hero_001 P5动画制作完成
大版本归档里程碑节点打 Tag：char/hero_001/v1.0便于回溯和对比
源文件存储PSD/Spine 文件用 Git LFS 管理二进制大文件不适合普通 Git
文件锁定编辑 .spine 文件前先 git lfs lock防止二进制文件合并冲突
废弃资源先标记 _deprecated_ 前缀，下个迭代删除留一个版本的缓冲期
