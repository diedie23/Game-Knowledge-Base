🔀 2. 动静分离原则

核心理念：将频繁变化的 UI 元素（动画、计时器、数值）与静态的 UI 元素（背景、边框、固定文字）分开放置，使引擎能独立缓存静态部分，减少重绘。

2.1 动/静分类标准

分类变化频率典型元素处理策略
静态不变 / 仅在打开时设一次面板背景、边框、标题文字、固定图标放入 Invalidation Box（UE5）/ Static Canvas（Unity），GPU 缓存
低频动态每秒 ≤ 1 次血量数字、金币数、等级可放入 Invalidation Box，仅在数据变化时 Invalidate
高频动态每帧 / 每秒多次计时器、动画、粒子、进度条动画必须隔离到独立层/Canvas，禁止与静态元素同层

🔴 铁律 #1：一个 Invalidation Box（或 Unity Canvas）内任何子元素变化，该 Box 内所有元素都重新绘制。因此高频动态元素绝对不能和静态元素放在同一个 Box 内。

🔴 铁律 #2：禁止在 Tick/Update 中修改不会变化的 UI 属性（如每帧设置文本为同一个值）。即使值不变，引擎仍会标记为脏(dirty)并触发重绘。

2.2 UE5 Invalidation Box 使用规范

  WBP_BattleHUD (Widget Blueprint)

      └── Widget_DamagePopup          // 伤害飘字

2.3 DrawCall 合批条件

条件合批结果说明
使用同一 Atlas 的相邻元素合批 ✅最理想情况
中间插入了不同 Atlas 的元素打断 ❌A-B-A 三个元素 = 3 次 DC
中间插入 Text 元素打断 ❌文字使用字体 Atlas，与 UI Atlas 不同
Material / Blend Mode 不同打断 ❌不同材质/混合模式无法合批
Clip / Mask 裁剪区域打断 ❌Clipping 产生额外 DC
⚠️ 合批优化技巧：① 同 Atlas 的图片在 UI 树中连续排列，不要被文字打断。② 文字元素集中放置，让引擎用一次 DC 渲染所有文字。③ 用 Retainer Box（UE5）将复杂子树渲染到离屏 RT，减少主 Pass DC。

🌳 3. UI 树层级扁平化

🔴 铁律：UI 树嵌套深度不得超过 10 层。每增加一层嵌套，布局计算复杂度近似 O(n)——10 层以上的深树在低端机上 Layout Pass 可能花费 > 2ms。

3.1 扁平化策略

问题反模式（嵌套过深）正确做法（扁平化）
多层装饰边框Border→Border→Border→Border 4层嵌套用 9 宫格单图实现 4 层边框效果，1 个 Image 节点
对齐布局HBox→VBox→HBox→VBox 交替嵌套用 Grid Panel（UE5）或 CSS Grid 一层搞定
间距Spacer→Spacer→Spacer 多个空节点使用父容器的 Padding/Slot.Padding
条件隐藏每个元素包一个 Visibility 容器直接控制元素自身的 Visibility，无需包装层
滚动列表ScrollBox 内放 100+ Widget使用 ListView（UE5）/ RecyclerView（Unity）虚拟列表

3.2 节点数预算分配

页面类型推荐节点数红线

普通页面（商城/背包）≤ 120200

✅❌ 4. 图文红黑榜

4.1 高频元素和静态元素混合 → 全页面重绘

❌ Don't — 倒计时器和背景面板在同一个 Canvas/Invalidation Box

[此处插入图片：Profiler 截图显示一个包含 40 个子元素的 Invalidation Box，因为一个 Timer Text 每帧变化，导致 40 个元素全部每帧重绘。Draw Call 40+]

后果：倒计时每帧变化 → 整个 Box 被标记为 dirty → 40 个子元素全部每帧重绘 → Draw Call 从 1（缓存态）膨胀到 40+。在低端机上直接掉帧。

✅ Do — 将倒计时器隔离到独立的高频层

[此处插入图片：Profiler 截图显示静态层 39 个元素 Draw Call = 1（缓存），高频层只有 1 个 Timer Text 每帧重绘。总 DC = 2]

方法：将 Timer 移出 Invalidation Box，放入独立的 Overlay 层。静态层GPU 缓存始终有效，仅 Timer 每帧重绘 1 次。

4.2 ScrollBox 内放 100+ Widget → 卡顿爆炸

❌ Don't — 好友列表 ScrollBox 内直接放 200 个 Widget

[此处插入图片：Widget Reflector 显示 ScrollBox 下 200 个子节点，内存占用 45MB，首次打开耗时 800ms]

后果：200 个 Widget 全部实例化（即使只有 10 个可见），内存爆炸，首次打开 > 500ms。滑动时 Layout Pass 计算 200 个节点位置，卡顿明显。

✅ Do — 使用 ListView 虚拟列表

[此处插入图片：同一好友列表使用 ListView，只有 12 个可见 Widget 实例被复用，内存 3MB，打开瞬间]

方法：UE5 ListView / Unity RecyclerView。只创建可见 + 缓冲区的 Widget 实例（约 12~15 个），滑动时复用而非创建。200 项数据只需 12 个 Widget。

❓ 6. 常见问题排查 FAQ

💬 Q1：UI 打开后帧率骤降 10+fps？

排查：① 运行 stat slate（UE5）检查 Draw Call 数。② 如果 DC > 50，用 Widget Reflector 找出打断合批的元素。③ 检查是否有高频元素在 Invalidation Box 内导致全量重绘。

修复：动静分离 + 合批优化 + Retainer Box 降频子树。

💬 Q2：长列表滑动时掉帧/白屏？

原因：ScrollBox 内直接放了所有 Widget，滑动时 Layout 计算量巨大。

修复：替换为 ListView（UE5）或 RecyclerView（Unity）。设置合适的缓冲区（OverscrollAmount / Extra Visible Count），确保快速滑动不白屏。

💬 Q3：Invalidation Box 设了但 DC 没减少？

原因①：Box 内有高频动态元素（如动画/Timer），导致缓存每帧失效。

原因②：在 Tick 中给不变的 UI 元素赋了相同的值（如 SetText(SameValue)），引擎仍标记为 dirty。

修复：① 高频元素移出 Box。② 赋值前加 if(NewValue != OldValue) 判断。③ 使用 Slate.InvalidationDebugging 1 可视化哪些元素导致了 Invalidation。

🤖 7. 如何提升 AI 生成 UE UMG 界面的准确率（Prompt 编写指南）

📌 背景：使用 ue-blueprint-tool 等 AI 辅助工具生成 UMG Widget Blueprint 时，Prompt 的结构化程度直接决定了产出物的工业级可用率。本章从 6 个核心维度给出可执行的编写规范，目标：一次生成 → 零手工修补 → 直接入工程。

7.1 明确的容器与层级结构（Hierarchy & Layout）

🔴 铁律：Prompt 中必须指明根节点类型。未指定根节点的 Prompt 会让 AI 随机选择 Canvas Panel / Overlay / User Widget，导致布局基础不可预测。

容器类型适用场景Prompt 关键词示例
Canvas Panel全屏 HUD、需要绝对坐标定位的界面"根节点为 Canvas Panel，支持绝对坐标布局"
Overlay元素需要层叠但不需精确坐标（弹窗遮罩层）"使用 Overlay 作为根节点，子元素堆叠排列"
Vertical Box线性垂直排列（设置列表、聊天消息流）"内容区使用 Vertical Box 依次排列各行"
Horizontal Box线性水平排列（底部按钮栏、Tab 导航）"底部按钮栏用 Horizontal Box，等间距排列 3 个按钮"
Grid Panel网格/背包/道具栏"物品列表使用 Grid Panel，4 列自动换行"
Size Box强制限制子元素尺寸（固定大小的图标/按钮）"每个道具格子用 Size Box 包裹，强制 80×80"
Scale Box等比缩放适配不同分辨率"最外层用 Scale Box 包裹，缩放策略 Scale to Fit"
Scroll Box可滚动内容（但长列表建议 ListView）"描述区域使用 Scroll Box，垂直滚动"
⚠️ 最佳实践：在 Prompt 开头先声明层级树，再描述每个节点的细节。例如先写"WBP_Popup 根节点为 Overlay，下辖：Image_BgMask → Vertical Box_Content → [标题/列表/按钮栏]"，再逐一描述各元素。这种"树形先行"写法比"自然语言平铺"的准确率高 40%+。

7.2 精准的锚点与对齐规范（Anchors & Alignment）

UE UMG 的布局系统核心是 Anchor（锚点）+ Alignment（对齐中心点）+ Offset（位置偏移）。三者的含义经常被混淆，导致 AI 生成的 UI 在分辨率变化时跑偏。

概念含义取值范围Prompt 中如何描述
Anchor元素挂靠在父容器的哪个位置（锚定参考点）Min(0,0)~Max(1,1)"锚点：居中(0.5,0.5)"、"锚点：左上角(0,0)"、"锚点：全屏填充(0,0)~(1,1)"
Alignment元素自身的对齐中心点偏移比例0~1"Alignment (0.5, 0.5) 以元素中心对齐锚点"
Position / Offset从锚点到元素实际位置的像素偏移px 值"Offset X=0, Y=-40（在锚点上方 40px 处）"

🔴 铁律 #1：描述居中元素时，必须同时写明 Anchor (0.5, 0.5) + Alignment (0.5, 0.5)。缺少 Alignment 会导致元素以左上角对齐锚点中心，视觉上偏右偏下。

🔴 铁律 #2：需要全屏自适应的元素（如背景遮罩、底图），使用 Anchor Min(0,0) Max(1,1) Offset 全 0，禁止使用绝对尺寸。

常用锚点 Preset 速查

Preset 名称Anchor Min/Max典型场景
Top-Left(0,0)~(0,0)HUD 左上角信息栏（等级/血量）
Top-Center(0.5,0)~(0.5,0)系统公告 Banner
Top-Right(1,0)~(1,0)Mini Map / 货币显示
Center(0.5,0.5)~(0.5,0.5)弹窗 / 对话框
Bottom-Center(0.5,1)~(0.5,1)操作按钮栏 / 技能槽
Fill(0,0)~(1,1)全屏遮罩 / 背景图
Horizontal-Fill + Top(0,0)~(1,0)顶部导航栏（水平铺满，垂直固定高度）

7.4 交互与可见性设置（Visibility & Hit Test）

UMG 的 Visibility 设置对 UI 性能和交互正确性影响巨大。未明确指定时，AI 往往将所有元素默认为 Visible（可见且响应点击），导致"幽灵点击拦截"——一个透明的背景图片意外拦截了下方按钮的点击事件。

Visibility 设置渲染响应点击适用元素Prompt 写法
Visible✅✅按钮、可交互列表项、输入框"Btn_Close: Visible"
Self Hit Test Invisible✅❌ 自身不响应
✅ 子元素响应		装饰性容器（内含可交互子元素）"VB_Content: Visibility = Self Hit Test Invisible"
Hit Test Invisible✅❌ 自身和所有子元素都不响应纯装饰层（背景图、边框、分割线）"Img_BgPanel: Visibility = Hit Test Invisible（纯装饰）"
Hidden❌（但占位）❌条件隐藏但保留布局空间"Txt_Error: Hidden（默认隐藏，逻辑控制显示）"
Collapsed❌（不占位）❌条件隐藏且不影响布局"VB_OptionalSection: Collapsed（按条件展开）"
⚠️ 防坑要点：
① 全屏半透明遮罩（如弹窗背景黑幕）如果需要点击关闭，设为 Visible；如果不需要点击关闭，必须设为 Hit Test Invisible，否则会拦截遮罩下方的所有点击。
② 布局容器（Vertical Box / Horizontal Box / Overlay）如果自身无交互，统一设为 Self Hit Test Invisible，仅让子元素中的按钮响应。
③ 纯装饰性 Image（图标、分割线、背景）统一 Hit Test Invisible，减少不必要的 Hit Test 遍历，提升 UI 交互性能。

7.7 Prompt 编写自检 Checklist

#检查项是否必填不填的后果
1是否指明了根节点类型（Canvas Panel / Overlay / ...）？必填AI 随机选根节点，布局不可控
2是否用树形结构描述了层级关系？必填嵌套关系混乱，需大量手动调整
3关键元素是否指定了 Anchor + Alignment？必填分辨率变化后 UI 跑偏
4每个控件是否使用了项目标准命名前缀？必填命名混乱，蓝图中难以查找
5需蓝图编程的控件是否标注了 Is Variable: True？必填蓝图中无法引用该控件
6是否区分了 Visible / Hit Test Invisible / Self Hit Test Invisible？强烈建议幽灵点击拦截，UI 交互异常
7是否提供了具体的 Padding / Margin / Font Size 数值？强烈建议间距不可预测，需手动微调
8是否描述了颜色值（Hex / RGBA）？强烈建议AI 使用默认白/黑色
9是否说明了多分辨率适配意图？建议可能在超宽屏下布局异常
10是否使用了 Size Box 限制关键尺寸？建议元素尺寸可能被父容器意外拉伸
