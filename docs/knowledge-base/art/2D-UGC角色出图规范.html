📐 1. 基础技术规范

1.1 画布与分辨率
要素规范说明
尺寸必须为 2 的 N 次幂常见：32×32、64×64、128×128、256×256、512×512、1024×1024、2048×2048
像素风格16×16 / 32×32 / 64×64 按需选择
高清风格推荐 512×512 或 1024×1024
长方形贴图允许非正方形但仍需满足 2^N，如 256×512
性能推荐口袋方舟推荐 256×256 作为性能最优基准
典型案例：Minecraft 基础皮肤贴图为 64×64 像素（旧版 64×32），标准 16x 像素网格结构。Roblox 缩略图推荐 1920×1080。
1.2 文件格式
格式适用场景
PNG（推荐）支持透明通道（Alpha），无损压缩，2D 角色首选格式
TGAPBR 流程使用，适合多通道信息的场景
JPEG仅用于不需要透明的展示图/缩略图
PSD/ASE源文件格式，不直接用于引擎
1.3 透明通道（Alpha）

🔴 铁律：角色轮廓外必须完全透明（Alpha = 0），导出 PNG 时确认保留 Alpha 通道。避免半透明像素残留在轮廓边缘。如果无透贴/自发光需求，不要包含 A 通道。

🧩 2. 角色部件拆分规范

2.1 模块化部件划分
部件层级包含内容说明
头发前发、后发、发饰需区分前后层级，支持独立换色
面部眼睛、眉毛、嘴巴、腮红独立小图叠加在面部底层上
身体/上装躯干、上衣、外套、披风需与下装在腰线处精确对接
下装裤子、裙子与上装的衔接线需点对点对齐
四肢/手套手臂、手套分长款/短款，不可超出预设范围
鞋子鞋、袜注意过膝/未过膝的归属规则
配饰帽子、眼镜、翅膀、尾巴等独立图层，灵活叠加
2.2 拆分对接核心要求
⛔ 硬性要求：衔接线必须精确对齐 · 所有部件共用同一原点坐标系 · 未被服装遮盖的皮肤需用基础裸模补全 · 层级（Z-Order）必须有统一编号规范
2.3 面数/复杂度控制（参考口袋方舟标准）
部件高精模型（展示用）低精模型（游戏内）
上衣 + 身体≤10,000 面≤2,500 面
手套 + 手≤3,500 面≤1,500 面
下装 + 腿≤4,000 面≤1,700 面
鞋子 + 脚≤3,500 面≤1,500 面
头发 + 发饰≤7,500~10,000 面≤2,500~3,000 面
总计≤30,000 面≤10,000 面

🎨 3. 美术风格与色彩规范

3.1 头身比与体型
风格头身比代表产品
Q 版圆润1:1 ~ 2:1蛋仔派对
Q 版人形2:1 ~ 3:1元梦之星、Gacha 系列
像素复古取决于精度Minecraft
写实卡通3:1 ~ 5:1部分 Roblox 体验
⚠️ 注意：确定头身比后全套部件必须统一，否则换装会出现比例失调。Q 版角色的头部通常占整体 40%~50%。
3.2 色彩规范
要素规范
主色数量单个角色建议 3~5 个主色 + 1~2 个点缀色
饱和度避免过高饱和（纯红、纯蓝），推荐中间调
明度对比不同部件之间保持足够明度差
色板统一提供统一调色板（Palette）
描边色统一使用比填充色深 20%~30% 的同色系色值
3.3 光影处理

🔴 统一光源方向：建议固定左上方 45° 或正上方光源，所有部件光影一致。像素风格使用 2~3 级明暗，卡通风格可使用赛璐珞式硬边阴影。

🎬 4. 动画与精灵图规范

4.1 精灵图（Sprite Sheet）
要素规范
排列方式推荐水平排列或网格排列，帧间距统一
帧尺寸所有帧保持完全相同的画布尺寸
帧间距（Padding）帧与帧之间预留 1~2px 间距
帧率像素动画 8~12 FPS；高清动画 12~24 FPS
每组动画帧数Idle 4~8 帧、Walk 6~8 帧、Run 6~10 帧、Jump 3~6 帧
4.2 动画兼容性
核心原则：所有部件必须共用同一套骨骼/关键帧 · 部件锚点在动画帧中位置一致 · 预留动画安全区（角色动作幅度大时确保精灵图画布足够大）

📁 5. 命名与资源管理规范

5.1 文件命名约定
推荐格式：类型_性别_部件_编号_变体

🔒 6. UGC 审核与合规注意事项

6.1 内容安全
⛔ 禁止内容：暴露/色情内容 · 暴力/血腥元素 · 敏感符号（宗教、种族歧视等）· 直接复制他人作品或知名 IP 角色
6.2 技术审核
检查项标准
文件大小限制单个资源 ≤ 1~5MB
格式验证仅接受 PNG/TGA
尺寸校验必须为 2^N 且不超过最大限制（通常 2048×2048）
透明通道检查背景区域 Alpha 必须为 0
部件兼容性测试新部件与所有现有基础部件无穿插、无错位

📱 7. 跨平台与性能优化

7.1 多端适配
平台注意事项
手机贴图尽量 ≤512×512，减少 Draw Call；精灵图合并 Atlas
PC可适当提高到 1024×1024
小程序/Web严格控制总包体，单图 ≤200KB；使用 WebP 压缩
7.2 性能优化建议
💡 优化要点：图集合并（Atlas）减少渲染批次 · 非像素风格资源开启 Mipmap · 确保纹理为 POT 尺寸 · 针对目标平台选择 ETC2/PVRTC/ASTC 压缩 · 远景使用低精度版本

🎯 10. 总结：核心设计原则

✅ 七大核心原则

1. 一致性优先 — 头身比、色板、光影方向、像素密度全局统一

2. 模块化设计 — 部件独立、接口标准化、支持自由组合

3. 性能敏感 — 移动端优先思维，严控贴图尺寸和文件大小

4. 安全区预留 — 动画、换装都需要额外的容错空间

5. 可扩展性 — 命名和目录结构支持持续扩充新部件

6. 合规底线 — 内容审核规则前置，避免返工

7. 跨端适配 — 从设计阶段就考虑多分辨率、多平台的适配方案

❌ 常见反面案例

· 不同部件使用不同画布尺寸 → 引擎拼合偏移

· 贴图使用非 POT 尺寸 → GPU 自动 padding 浪费显存

· 不统一光源方向 → 换装后阴影冲突

· Mask 图使用 JPEG → 通道噪点导致换色边缘脏色

· 未预留 Padding → 动画时关节处缝隙/穿模

· 命名随意 → 自动化脚本无法匹配
