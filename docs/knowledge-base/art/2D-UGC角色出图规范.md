# 2D UGC 角色出图规范与注意事项

> 基于市面主流 UGC 游戏（Minecraft / 我的世界、蛋仔派对、元梦之星、口袋方舟 Ark Online、Roblox、Gacha 系列、迷你世界等）的实践经验提炼。
>
> **文档版本**：v1.0 | **更新日期**：2026-03-31 | **维护人**：美术组

---

## 一、基础技术规范

### 1.1 画布与分辨率

| 要素 | 规范说明 |
|------|----------|
| **尺寸必须为 2 的 N 次幂** | 常见尺寸：32×32、64×64、128×128、256×256、512×512、1024×1024、2048×2048 |
| **像素风格** | 16×16 / 32×32 / 64×64 按需选择；像素越大细节越丰富，但风格一致性要求越高 |
| **高清风格** | 推荐 512×512 或 1024×1024；过大会增加包体与内存开销 |
| **长方形贴图** | 允许非正方形但仍需满足 2^N，如 256×512、512×1024 |
| **性能推荐** | 口袋方舟推荐 256×256（±10%）作为性能最优基准 |

**典型案例**：
- **Minecraft / 我的世界**：基础皮肤贴图为 **64×64** 像素（旧版 64×32），标准 16x 像素网格结构
- **Roblox**：缩略图推荐 **1920×1080**（16:9），资产图片 **PNG/JPEG**，≤50MB

### 1.2 文件格式

| 格式 | 适用场景 |
|------|----------|
| **PNG**（推荐） | 支持透明通道（Alpha），无损压缩，2D 角色首选格式 |
| **TGA** | PBR 流程使用，适合需要多通道信息的场景（金属度/粗糙度/AO/自发光） |
| **JPEG** | 仅用于不需要透明的展示图/缩略图 |
| **PSD/ASE** | 源文件格式，用于工作流交接，不直接用于引擎 |

### 1.3 透明通道（Alpha）

- 角色轮廓外的区域**必须为完全透明**（Alpha = 0），否则会出现白边/黑边
- 导出 PNG 时需确认**保留 Alpha 通道**
- 避免半透明像素残留在轮廓边缘（尤其手动擦除背景时）
- 如果无透贴/自发光需求，**不要包含 A 通道**，避免不必要的资源开销

## 二、角色部件拆分规范

### 2.1 模块化部件划分

2D UGC 角色通常采用**分层/模块化**设计，以支持换装和自定义。常见拆分方式：

| 部件层级 | 包含内容 | 说明 |
|----------|----------|------|
| **头发** | 前发、后发、发饰 | 需区分前后层级，支持独立换色 |
| **面部** | 眼睛、眉毛、嘴巴、腮红 | 通常为独立小图叠加在面部底层上 |
| **身体/上装** | 躯干、上衣、外套、披风 | 需与下装在腰线处精确对接 |
| **下装** | 裤子、裙子 | 与上装的衔接线需点对点对齐 |
| **四肢/手套** | 手臂、手套 | 分长款/短款，不可超出预设范围 |
| **鞋子** | 鞋、袜 | 注意过膝/未过膝的归属规则 |
| **配饰** | 帽子、眼镜、翅膀、尾巴等 | 独立图层，灵活叠加 |

### 2.2 拆分对接核心要求

- **衔接线必须精确对齐**：各部件在拼合处的边缘像素需严格对齐，避免出现缝隙或错位
- **统一锚点/原点**：所有部件共用同一原点坐标系，确保换装时位置不偏移
- **裸模补齐**：未被服装遮盖的皮肤部分需用基础裸模补全（口袋方舟规范）
- **层级（Z-Order）明确定义**：前后遮挡关系需有统一的层级编号规范
- **镜像对称处理**：Minecraft 单层皮肤中，一侧手臂/腿会自动镜像；需注意不对称设计的兼容性

### 2.3 面数/复杂度控制（参考口袋方舟标准）

| 部件 | 高精模型（展示用） | 低精模型（游戏内） |
|------|-------------------|-------------------|
| 上衣 + 身体 | ≤10,000 面 | ≤2,500 面 |
| 手套 + 手 | ≤3,500 面 | ≤1,500 面 |
| 下装 + 腿 | ≤4,000 面 | ≤1,700 面 |
| 鞋子 + 脚 | ≤3,500 面 | ≤1,500 面 |
| 头发 + 发饰 | ≤7,500~10,000 面 | ≤2,500~3,000 面 |
| **总计** | **≤30,000 面** | **≤10,000 面** |

> 对于纯 2D 游戏可类比为"单个部件精灵图的像素复杂度"——控制每个部件的精灵图尺寸和帧数。

## 三、美术风格与色彩规范

### 3.1 头身比与体型

| 风格 | 头身比 | 代表产品 |
|------|--------|----------|
| Q 版圆润 | 1:1 ~ 2:1 | 蛋仔派对（球形身体） |
| Q 版人形 | 2:1 ~ 3:1 | 元梦之星、Gacha 系列 |
| 像素复古 | 取决于精度 | Minecraft |
| 写实卡通 | 3:1 ~ 5:1 | 部分 Roblox 体验 |

**注意点**：
- 确定头身比后**全套部件必须统一**，否则换装会出现比例失调
- Q 版角色的头部通常占整体 40%~50%，需给面部表情留足像素空间

### 3.2 色彩规范

| 要素 | 规范 |
|------|------|
| **主色数量** | 单个角色建议 3~5 个主色 + 1~2 个点缀色 |
| **饱和度** | 避免过高饱和（纯红、纯蓝），推荐使用中间调，便于加阴影/高光 |
| **明度对比** | 不同部件之间保持足够明度差，确保在小尺寸下仍可辨识 |
| **色板统一** | 提供统一调色板（Palette），确保所有 UGC 创作者的角色风格协调 |
| **肤色范围** | 需预设多样化肤色选项，避免过暗或过亮的极端值 |
| **描边色** | 如有描边，统一使用比填充色深 20%~30% 的同色系色值 |

### 3.3 光影处理

- **像素风格**：通常使用 2~3 级明暗（亮面 → 固有色 → 暗面），不做平滑渐变
- **卡通风格**：可使用赛璐珞式硬边阴影或简单渐变
- **统一光源方向**：建议固定左上方 45° 或正上方光源，所有部件光影一致
- **避免极端对比**：布料类明暗对比不宜过强，保持柔和过渡

## 四、动画与精灵图规范

### 4.1 精灵图（Sprite Sheet）

| 要素 | 规范 |
|------|------|
| **排列方式** | 推荐水平排列或网格排列，帧间距统一 |
| **帧尺寸** | 所有帧保持完全相同的画布尺寸 |
| **帧间距（Padding）** | 帧与帧之间预留 1~2px 间距，防止采样溢出 |
| **帧率** | 像素动画常用 8~12 FPS；高清动画 12~24 FPS |
| **每组动画帧数** | Idle（4~8帧）、Walk（6~8帧）、Run（6~10帧）、Jump（3~6帧） |

### 4.2 动画兼容性

- **所有部件必须共用同一套骨骼/关键帧**：换装后动画不能错位
- **部件锚点在动画帧中位置一致**：帽子、头发等附属物需跟随头部运动
- **预留动画安全区**：角色动作幅度大时（如跳跃、挥手），需确保精灵图画布足够大，不会裁切

## 五、命名与资源管理规范

### 5.1 文件命名约定

推荐格式：`类型_性别_部件_编号_变体`

```
示例：
SK_Female_Body_001_Show    // 女性上装-编号001-展示用
SK_Male_Hair_012           // 男性头发-编号012-游戏内
SP_Unisex_Hat_003_Red      // 通用帽子-编号003-红色变体
```

### 5.2 材质球命名（参考口袋方舟）

| 部件 | 材质名 | 透贴 |
|------|--------|------|
| 上装 | `ClothUpper` | `ClothUpper_Mask` |
| 下装 | `ClothLower` | `ClothLower_Mask` |
| 手套 | `Glove` | `Glove_Mask` |
| 鞋子 | `ClothShoes` | `ClothShoes_Mask` |
| 身体 | `Body` | — |
| 头发 | `Hair` | — |
| 发饰 | `HairAcc` | `HairAcc_Mask` |

### 5.3 目录结构建议

```
/Characters
  /Base         // 基础裸模
  /Hair         // 头发
  /Face         // 面部表情
  /UpperBody    // 上装
  /LowerBody    // 下装
  /Shoes        // 鞋子
  /Accessories  // 配饰
  /Thumbnails   // 缩略图/预览图
```

## 六、UGC 审核与合规注意事项

### 6.1 内容安全

- **禁止暴露/色情内容**：角色服装需有最低覆盖标准
- **禁止暴力/血腥元素**：武器、伤痕等需符合目标年龄分级
- **禁止敏感符号**：避免包含政治、宗教、种族歧视等敏感内容
- **版权合规**：UGC 创作不得直接复制他人作品或知名 IP 角色

### 6.2 技术审核

- **文件大小限制**：单个资源通常需控制在 **1~5MB** 以内
- **格式验证**：仅接受指定格式（PNG/TGA），拒绝非标准格式
- **尺寸校验**：必须为 2^N 且不超过最大限制（通常 2048×2048）
- **透明通道检查**：背景区域 Alpha 必须为 0
- **部件兼容性测试**：新部件需与所有现有基础部件无穿插、无错位

## 七、跨平台与性能优化

### 7.1 多端适配

| 平台 | 注意事项 |
|------|----------|
| **手机** | 贴图尽量 ≤512×512，减少 Draw Call；精灵图合并 Atlas |
| **PC** | 可适当提高到 1024×1024，支持更多细节 |
| **小程序/Web** | 严格控制总包体，单图 ≤200KB 为佳；使用 WebP 压缩 |

### 7.2 性能优化建议

- **图集合并（Atlas）**：将同一角色的所有部件打入一张图集，减少渲染批次
- **Mipmap**：对非像素风格资源开启 Mipmap，提升远距离显示效果
- **POT（Power of Two）**：确保纹理为 2 的幂次尺寸，兼容更多 GPU
- **压缩格式**：针对目标平台选择 ETC2（Android）/ PVRTC（iOS）/ ASTC（通用）
- **LOD（细节层级）**：远景使用低精度版本，口袋方舟提供了"局外/局内"两套精度

## 八、出图检查清单（Checklist）

### 发布前必查项

- [ ] 画布尺寸为 2 的 N 次幂
- [ ] 文件格式为 PNG（含正确 Alpha 通道）
- [ ] 所有部件统一原点和锚点
- [ ] 各部件衔接处无缝隙、无错位
- [ ] 层级（Z-Order）正确，无异常遮挡
- [ ] 色彩饱和度适中，无极端色值
- [ ] 光源方向统一
- [ ] 动画帧尺寸一致，帧间距正确
- [ ] 换装兼容性测试通过（与所有基础部件组合无穿插）
- [ ] 文件大小在限制范围内
- [ ] 命名符合约定规范
- [ ] 无版权/合规风险内容
- [ ] 多端显示测试通过（手机/PC/Web）

## 九、各平台关键差异速查

| 平台 | 核心特点 | 角色出图关键规范 |
|------|----------|-----------------|
| **Minecraft** | 像素风格，UV展开式贴图 | 64×64 PNG，固定UV布局，单/双层结构 |
| **蛋仔派对** | 圆形Q版，3D渲染但2D展示 | 极简造型，统一球形体型，配饰为主要差异化 |
| **元梦之星** | 人形Q版，3D为主 | 人型小人，联动 IP 多，服装拆分较细 |
| **口袋方舟** | 二次元/卡通，完整换装系统 | 五大部件拆分（上装/下装/鞋/手套/头发），严格面数和UV规范 |
| **Roblox** | 多风格混合，3D + 2D纹理 | 缩略图 1920×1080，PNG/JPEG ≤50MB，自由度极高 |
| **Gacha 系列** | 纸娃娃式2D换装 | 分层叠加，部件极细（100+可自定义项），统一画风 |
| **迷你世界** | 类MC像素风 | 类似MC的UV贴图结构，像素化设计 |

## 十、总结：核心设计原则

1. **一致性优先**：头身比、色板、光影方向、像素密度全局统一
2. **模块化设计**：部件独立、接口标准化、支持自由组合
3. **性能敏感**：移动端优先思维，严控贴图尺寸和文件大小
4. **安全区预留**：动画、换装都需要额外的容错空间
5. **可扩展性**：命名和目录结构支持持续扩充新部件
6. **合规底线**：内容审核规则前置，避免返工
7. **跨端适配**：从设计阶段就考虑多分辨率、多平台的适配方案
