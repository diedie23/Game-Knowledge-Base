🚨 1. 性能红线与切图规格

⛔ 红线 = 强制标准。不满足以下条件的切图资产将被 CI/CD 脚本自动拦截。

📌 流程约定 #1 偶数尺寸原则：所有 UI 切图的宽度和高度必须为偶数（能被 2 整除）。奇数尺寸在纹理压缩（ETC2/ASTC/BC7）时会被 pad 到偶数，导致边缘出现 1px 模糊条纹。推荐 4 的整数倍（ASTC 4×4 最小块）。

📌 流程约定 #2 最大尺寸限制：单张切图 ≤ 2048×2048。超大 UI 元素必须拆分为多个切片或使用 9 宫格拉伸。

📌 流程约定 #3 透明通道审计：不含透明区域的 UI 元素（如纯色背景板）禁止导出 Alpha 通道——用 RGB 24bit 代替 RGBA 32bit，节省 25% 内存。

✅❌ 3. 图文红黑榜

3.1 奇数尺寸 → 压缩模糊

❌ Don't — 切图尺寸为奇数（如 127×63）

[此处插入图片：按钮切图 127×63px，导入引擎后 ETC2 压缩将其 pad 到 128×64，右侧和底部各出现 1px 的半透明模糊条纹]

后果：ETC2/ASTC 等块压缩算法以 4×4 为最小块。奇数尺寸被引擎自动 pad，右边缘和底边缘出现 1px 模糊条，在纯色/渐变按钮上尤为明显。移动端必现。

✅ Do — 所有切图宽高为 4 的整数倍

[此处插入图片：同一按钮调整为 128×64px，压缩后边缘清晰无模糊，与设计稿完全一致]

方法：Photoshop 切图后，图像 → 画布大小，将宽高调为最近的 4 的整数倍。多出的 1~3px 用透明像素填充。批量处理可用 TexturePacker 的 Force Squared + Size Constraints 配置。

3.2 命名混乱 → 状态丢失

❌ Don't — 随意命名（中文/空格/无后缀）

[此处插入图片：资源管理器中一堆文件如"开始按钮.png"、"按钮-按下(1).png"、"btn copy 2.png"——无法识别状态]

后果：① 程序无法自动匹配按钮的 N/P/D 状态，需手动逐个配置。② 中文/空格在部分构建管线中会导致路径解析失败。③ 团队协作时无人知道哪个文件对应哪个状态。

✅ Do — 严格遵循命名公式 + 状态后缀

[此处插入图片：规范命名的文件列表 T_UI_Login_BtnStart_N/P/D.png，整齐排列，一目了然]

方法：T_UI_{Module}_{Element}_{State}.png。程序可通过正则自动绑定同名不同后缀的状态图。全英文、无空格、下划线分隔。

❓ 5. 常见问题排查 FAQ

💬 Q1：按钮在手机上右侧/底部有 1px 杂线？

原因：切图尺寸为奇数，ETC2/ASTC 压缩时 pad 了透明像素。

修复：将切图宽高调整为 4 的整数倍。可用批处理脚本：convert input.png -extent WxH output.png（ImageMagick）。

💬 Q2：按钮点击时没有切换到按下态图片？

原因①：文件命名后缀不一致。程序代码查找 _P 后缀，而美术命名为 _pressed 或 _按下。

原因②：按下态图片没有交付——美术只出了 _N 一张。

修复：严格使用本规范的后缀字典。每个可交互元素至少提供 _N 和 _D 两态，按钮类加 _P。

💬 Q3：UI Atlas 单张超过 2048 导致低端机卡顿？

原因：将所有 UI 元素合入一张巨型 Atlas（如 4096×4096），低端 GPU 显存不够导致频繁换入换出。

修复：按功能模块拆分 Atlas（Login、Shop、Battle 各一张），每张 ≤ 2048×2048。不同页面的 UI 不共用 Atlas，避免跨页面纹理常驻。

7.3 Figma Export Plugin 核心逻辑
// figma-ui-export-plugin (TypeScript)
// 核心功能：遍历选中的 Component Set，按 Variant 导出并规范命名

interface ExportConfig {
  prefix: "T_UI";
  module: string;          // 从父 Frame 名称推断
  scale: 1;
  format: "PNG";
  forceEvenSize: true;     // 强制偶数尺寸
}

async function exportVariants(componentSet: ComponentSetNode, config: ExportConfig) {
  const variants = componentSet.children as ComponentNode[];
  
  for (const variant of variants) {
    // 从 Variant property 提取状态后缀
    const state = variant.variantProperties?.["State"] || "N";
    const stateMap: Record<string, string> = {
      "Normal": "N", "Pressed": "P", "Disabled": "D",
      "Hover": "H", "Selected": "S", "Focused": "F"
    };
    const suffix = stateMap[state] || state;
    
    // 构建规范文件名
    const element = componentSet.name.replace(/\s+/g, "");
    const fileName = `${config.prefix}_${config.module}_${element}_${suffix}`;
    
    // 尺寸校正为偶数
    let w = Math.ceil(variant.width);
    let h = Math.ceil(variant.height);
    if (w % 2 !== 0) w += 1;
    if (h % 2 !== 0) h += 1;
    
    // 导出
    const bytes = await variant.exportAsync({ format: "PNG", constraint: { type: "SCALE", value: 1 } });
    // ... 保存为 fileName.png，尺寸后处理
  }
}
7.4 Post-Process 批处理脚本
# post_process_slices.py — 导出后自动化处理
from PIL import Image
from pathlib import Path
import re

NAMING_REGEX = re.compile(r'^T_UI_[A-Z][a-zA-Z]+_[A-Z][a-zA-Z0-9]+_(N|P|H|D|S|F|On|Off|Fill|Bg|9s|Mask)\.png$')

def process_directory(input_dir: str, output_dir: str):
    results = {"ok": 0, "fixed": 0, "error": 0}
    
    for png in Path(input_dir).glob("*.png"):
        # 1. 命名校验
        if not NAMING_REGEX.match(png.name):
            print(f"[ERROR] 命名不合规: {png.name}")
            results["error"] += 1
            continue
        
        # 2. 偶数尺寸校正
        img = Image.open(png)
        w, h = img.size
        new_w = w if w % 4 == 0 else (w // 4 + 1) * 4
        new_h = h if h % 4 == 0 else (h // 4 + 1) * 4
        
        if (new_w, new_h) != (w, h):
            # 扩展画布，透明填充
            new_img = Image.new("RGBA", (new_w, new_h), (0, 0, 0, 0))
            new_img.paste(img, (0, 0))
            img = new_img
            results["fixed"] += 1
        
        # 3. 输出
        img.save(Path(output_dir) / png.name, optimize=True)
        results["ok"] += 1
    
    print(f"✅ 处理完成: {results['ok']} OK, {results['fixed']} 尺寸修正, {results['error']} 命名错误")

8.3 Pre-commit Hook（本地即时反馈）
#!/bin/bash
# .git/hooks/pre-commit — UI 切图本地预检

UI_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '^Content/UI/.*\.png$')

if [ -z "$UI_FILES" ]; then
    exit 0
fi

echo "🔍 检查 UI 切图规范..."
ERRORS=0

for file in $UI_FILES; do
    # 快速命名检查
    basename=$(basename "$file")
    if ! echo "$basename" | grep -qE '^T_UI_[A-Z][a-zA-Z]+_[A-Z][a-zA-Z0-9]+_(N|P|H|D|S|F|On|Off|Fill|Bg|9s|Mask)\.png$'; then
        echo "❌ 命名不合规: $basename"
        ERRORS=$((ERRORS + 1))
    fi
    
    # 快速尺寸检查 (需要 identify 命令 from ImageMagick)
    SIZE=$(identify -format '%w %h' "$file" 2>/dev/null)
    W=$(echo $SIZE | cut -d' ' -f1)
    H=$(echo $SIZE | cut -d' ' -f2)
    if [ $((W % 2)) -ne 0 ] || [ $((H % 2)) -ne 0 ]; then
        echo "❌ 尺寸非偶数: $basename (${W}x${H})"
        ERRORS=$((ERRORS + 1))
    fi
done

if [ $ERRORS -gt 0 ]; then
    echo "⛔ 发现 $ERRORS 个问题，请修复后重新提交"
    exit 1
fi

echo "✅ UI 切图校验通过"
⚠️ 注意：Pre-commit Hook 仅做轻量快速检查（命名+尺寸），完整校验（重复检测、Atlas 归属）在 CI 服务器端执行，避免本地 commit 过慢。

📊 9. PM视角：UI 资产管理 KPI 看板

PM 需要通过数据驱动来管控 UI 切图资产的质量、效率和成本。以下 KPI 体系覆盖生产效率、质量合规、资产健康度三个维度。

9.1 核心 KPI 定义
维度指标名称公式目标值数据源
质量命名合规率合规文件数 / 总 UI 文件数 × 100%≥ 99%CI 命名检查日志
尺寸合规率偶数尺寸文件数 / 总数 × 100%= 100%CI 尺寸检查日志
CI 首次通过率一次提交即通过 / 总提交数 × 100%≥ 85%CI Build History
效率切图交付周期设计定稿 → 切图入库 平均耗时≤ 4hTAPD 工单时间戳
自动化覆盖率自动导出数 / 总切图数 × 100%≥ 70%Figma 插件日志
返工率需要重新切图的次数 / 总交付数 × 100%≤ 10%TAPD 退回工单
资产健康Atlas 填充率已用像素面积 / Atlas 总面积 × 100%≥ 75%TexturePacker 报告
冗余资产率SSIM>0.98 重复贴图数 / 总数 × 100%≤ 3%重复检测脚本
内存预算遵守率实际 UI 内存 / 预算内存 × 100%≤ 100%引擎 Profiler
9.2 看板可视化维度
看板区域展示内容可视化形式刷新频率
总览仪表盘9 项 KPI 当前值 + 趋势大数字卡片 + 趋势图实时
模块资产统计各模块切图数量、大小、Atlas 使用堆叠柱状图每日
违规热力图各规则触发次数 × 模块分布热力矩阵每周
交付效率趋势切图周期 + 返工率走势双轴折线图每周
Top 问题资产违规次数 Top10 + 未修复列表排行榜表格每日
内存预算仪表盘各平台 UI 内存使用量 vs 预算环形进度条每次构建
9.3 资产生命周期管理
阶段PM 关注点自动化动作
创建是否遵循规范？是否有已存在的可复用资产？Figma 插件导出 + 重复检测
使用中被多少界面引用？是否合入正确 Atlas？引用计数统计 + Atlas 分析
闲置超过 30 天未被引用的资产自动标记 + 清理候选通知
废弃确认不再需要的资产移入 Deprecated 目录 → 下版本删除
9.4 质量门禁联动
里程碑切图资产质量门禁不达标处置
Alpha命名合规率 ≥ 95%；尺寸合规率 ≥ 98%记录债务，不阻断
Beta命名合规率 ≥ 99%；冗余率 ≤ 5%；Atlas 填充率 ≥ 70%延迟发布直到达标
RC全部 Error 级规则通过；内存预算遵守率 ≤ 100%阻断发布
Live 热更增量切图全部通过 CI 校验回退热更
✅ 最佳实践：将 CI 报告自动推送至团队 IM 群组，每周五 PM 生成「UI 资产健康周报」包含 KPI 趋势 + Top 问题 + 下周改进计划。
