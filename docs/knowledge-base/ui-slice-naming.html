✂️ UI 切图与命名规范 UI 规范

面向小游戏/休闲游戏场景的 UI 资源切图标准与统一命名规则。帮助美术团队和外包 CP 减少因命名混乱、格式错误导致的返工问题。

📂 UI · 界面 🎯 适用：小游戏 / 休闲游戏 / 轻量级项目 👥 受众：UI美术 / 外包CP / APM 📅 更新：2026-04

📝 一、统一命名规则

💼
APM 视角：命名不规范是外包交付中返工率最高的问题之一。据统计，约 30% 的资源返工源于命名不合规——包括中文路径、空格、大小写混乱、缺少状态后缀等。制定统一命名规则并在合同中明确写入，能节省大量校验时间。

1.1 核心命名模式

通用 UI 资源命名格式
模块_类型_名称_状态.png 📋
📦 模块 — 所属功能页面 🏷️ 类型 — 控件类型 📝 名称 — 具体描述 🔄 状态 — 交互状态
示例：
Login_Btn_Start_Normal.png — 登录页 / 按钮 / 开始 / 正常态
Shop_Icon_Coin_Gold.png — 商城页 / 图标 / 金币 / 金色
Battle_Bg_Sky_Night.png — 战斗页 / 背景 / 天空 / 夜晚
Main_Tab_Home_Selected.png — 主页 / 标签 / 首页 / 选中态

1.2 各字段取值规范

字段取值范围说明
模块（Module） Login Main Shop Battle Setting Common Lobby Chat Rank Result 所属功能页面/模块名，首字母大写
类型（Type） Btn Bg Icon Text Bar Panel Tab Popup Deco Avatar Frame Mask Spine 控件/资源类型缩写，首字母大写
名称（Name） 自由命名，驼峰式 具体功能描述，如 Start Close GoldCoin
状态（State） Normal Pressed Disabled Selected Hover Locked Glow 交互状态后缀（无状态可省略）
⚠️ 外包常见错误
  • 中文文件名：登录_开始按钮.png → 导入引擎会乱码/路径失效
  • 含空格：Login Btn Start.png → 程序引用出错
  • 缺状态后缀：只给一个 Login_Btn_Start.png，不区分 Normal/Pressed
  • 大小写不一致：login_btn_start.png vs Login_Btn_Start.png 混用
  • 编号代替语义名：UI_001.png UI_002.png → 无法辨别用途
  • 特殊字符：Shop【金币】icon.png → 含中文标点和括号

1.3 不同引擎的命名惯例

引擎推荐前缀/习惯注意事项
Unity 无硬性前缀，按 模块/类型 文件夹分组 路径区分大小写（Android 端），避免重名
UE T_（贴图）M_（材质）WBP_（Widget） 遵循 Epic 前缀约定，Texture 用 T_模块_名称
Cocos 无硬性规定，按目录分类 UUID 引用，改名后需检查引用是否丢失
微信小游戏 无前缀要求，全小写+下划线更安全 包体大小敏感，文件名短一些节约字节

✂️ 二、切图标准

2.1 切图基本要求

要求说明常见错误
尺寸对齐 宽高建议为偶数（最好 4 的倍数），部分引擎纹理压缩要求 4×4 block 切出 337×251 的奇数尺寸，纹理压缩出错
去除多余透明 裁切到内容边界，去除大面积空白 切图保留了大量透明区域浪费内存
保留安全边距 内容周围保留 1~2px 透明边距，防止图集合并后出血 紧贴像素切图，打进图集后边缘出现色线
统一画布 同一组件的不同状态应保持画布尺寸一致 Normal 120×40、Pressed 118×39，导致按钮闪跳
独立图层导出 每个可复用/可换色的元素应单独切图 把按钮文字和按钮底图合并到一张切图里
⚠️ 外包常见错误
  • 合并切图：把多个独立UI元素合到一张大图交付，程序无法分别控制
  • PSD直出：直接导出整个PSD的合并图层，不分图切割
  • 尺寸随意：不关注尺寸是否为偶数/4的倍数，纹理压缩后出现绿边
  • 遮挡残留：切图时隐藏图层不彻底，残留了其他元素像素
  • 色彩模式错误：以CMYK模式导出，导入引擎后颜色偏差
❌ BEFORE — 不规范切图
  • 按钮底图+文字合为一张图
  • 大量透明区域未裁切
  • 不同状态尺寸不一致
  • 奇数像素尺寸（337×251）
  • 9宫格拉伸区域被合并
✅ AFTER — 规范切图
  • 底图与文字分离，程序可独立控制
  • 裁切到内容边缘 + 1px 安全边距
  • 所有状态统一 120×40 画布
  • 尺寸为 4 的倍数（120×40）
  • 9宫格元素单独切图并标注拉伸区域

2.2 9宫格切图特别注意

⚠️
9宫格（Nine-Slice/Nine-Patch）资源必须单独切图，不能与其他元素合并。切图时确保四边可拉伸区域是纯色或可重复的纹理，否则拉伸后会变形。详见 9宫格与颜色空间规范。

📦 三、输出格式选择

不同场景需要不同的文件格式。选错格式会导致透明度丢失、色彩偏移或文件体积过大。

PNG
Portable Network Graphics
无损压缩 + Alpha 通道
适用：UI 切图首选，按钮、图标、面板等所有需要透明的资源
首选格式
PSD
Photoshop Document
源文件/工作文件
适用：交付源文件，保留图层结构，便于二次修改
源文件
TGA
Targa Image
无压缩 + Alpha
适用：UE 项目贴图，需要无损精度且引擎可内部压缩
UE 首选
FBX
Filmbox Format
3D 模型/骨骼动画
适用：3D UI 元素、Spine 导出后的骨骼动画数据
3D 模型
JPG
JPEG Image
有损压缩，无 Alpha
适用：仅用于全屏背景图且无需透明时，可节省 50%+ 体积
仅限背景
WebP
Web Picture
支持 Alpha + 高压缩率
适用：小游戏/Web 平台，体积仅为 PNG 的 40~60%
小游戏优选
💼
APM 管理建议：在外包合同中明确规定交付格式。常见约定：
① 切图交付 = PNG（带 Alpha）
② 源文件交付 = PSD（保留图层）
③ UE 项目 = TGA + PSD 源文件
④ 小游戏项目 = WebP + PNG 双格式
⚠️ 外包常见错误
  • 全用 JPG：按钮、图标也用 JPG 交付，Alpha 通道全部丢失
  • 不交源文件：只给切图 PNG，不给 PSD 源文件，后续无法修改
  • BMP 交付：使用 BMP 格式，无 Alpha 且文件巨大
  • 色彩空间错误：导出时未转换为 sRGB，颜色在引擎中偏差严重

🔄 六、Before / After 对比总结

❌ BEFORE — 无规范时
  • 文件命名混乱：按钮1.png UI 002.png aaa_final.png
  • 切图尺寸随意，各状态大小不一
  • 全用 JPG 交付，缺少透明通道
  • 文件散落在根目录，无分类
  • 9宫格和普通切图混在一起
  • 每次交付花 2~3 小时校验返工
  • 外包反复沟通修改标准
✅ AFTER — 执行规范后
  • 统一 模块_类型_名称_状态.png 命名，一目了然
  • 切图尺寸统一、偶数像素、有安全边距
  • 格式按需：PNG/TGA/WebP 明确分工
  • 模块化目录结构，快速定位资源
  • 9宫格独立目录，标注拉伸区域
  • 交付校验 10 分钟完成
  • 外包合同附带 Checklist，一次通过率 90%+
