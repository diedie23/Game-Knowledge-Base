1️⃣ 术语速查：5 大高频词深度解析

以下 5 个词是美术、程序、TA 开会时的「口头禅」。读完这一节，你就能在评审会上听懂 80% 的性能讨论。

📞 Drawcall / Batching

Draw Call（绘制调用） & Batching（合批）

CPU 瓶颈
帧率杀手 Top1

🔍 一句话解释

每次 CPU 告诉 GPU「画这个东西」，就产生一次 Drawcall。想象你是工地包工头（CPU），每画一堵墙都要打一个电话给工人（GPU）。电话越多，包工头越累 → 游戏越卡。Batching（合批）就是把多个电话合成一个："把这 50 堵墙一起画了！"

⚡ 对性能的影响

每个 Drawcall 有 固定 CPU 开销（约 0.01~0.05ms）

移动端同屏 Drawcall 超过 200~300 就可能导致 CPU 瓶颈

最常见原因：材质太多（不同材质 = 无法合批）、图集未合并

🔧 常见优化手段

Static Batching：静态物体预合并（空间换时间）

Dynamic Batching：动态小物件自动合批

GPU Instancing：相同模型批量绘制（草、树）

SRP Batcher：URP/HDRP 自动材质合批

图集合并（Atlas）：2D/UI 的终极合批方案

💰 预算管控：项目启动时应和 TA 确定 Drawcall 上限（通常 ≤200），写入美术规范

⏰ 里程碑检查：每个版本要跑 RenderDoc / Xcode GPU Debugger 检查实际值

🚨 风险信号：如果 TA 说"合批率很低"或"材质球太多了"，意味着可能要重做资源，需评估排期影响

📊 你可以这样问："这个场景现在 Drawcall 多少？合批率有百分之多少？"

🎨 Overdraw

Overdraw（过度绘制 / 像素重绘）

GPU 瓶颈
发热元凶

🔍 一句话解释

屏幕上同一个像素被重复涂了好几遍颜色。想象你在墙上刷油漆，先刷白色，再刷蓝色，最后刷红色 —— 前两遍全浪费了。Overdraw 就是 GPU 做的这种"无用功"。

⚡ 对性能的影响

Overdraw 2x = GPU 干两倍活 → 掉帧 + 发热

移动端 GPU 填充率有限，全屏 Overdraw ≤2.5x 是安全线

重灾区：半透明特效叠加、UI 层级过深、2D 大面积粒子

🔧 常见优化手段

特效控制粒子数量和面积，避免全屏铺满

UI 避免大量透明层重叠（不可见区域及时隐藏）

不透明物体从前往后排序（Early-Z 剔除）

用特效预算表限制单技能最大粒子数

🔥 关注场景：当玩家反馈"一放大招就卡"，八成是 Overdraw 爆了

📱 机型敏感：低端机 GPU 填充率差，Overdraw 影响翻倍

🚨 风险信号：特效同学说"效果没法再砍了"、"层数至少要 5 层"，你需要升级到 TA 评估

📊 你可以这样问："这套特效的 Overdraw 倍率是多少？在低端机实测过吗？"

🗜️ 贴图压缩 (ASTC / ETC2)

Texture Compression — ASTC / ETC2 / PVRTC

内存占用
包体大小

🔍 一句话解释

原始贴图像高清照片（几十 MB），游戏里要压缩成 GPU 能直接读取的格式。就像视频有 .mp4，图片有 .jpg，游戏贴图也有自己的"压缩格式"。ASTC 和 ETC2 是移动端最常用的两种。

📊 主流格式对比

格式 | 平台 | 质量 | 大小(1024²) |

ASTC 4x4 | iOS / 高端安卓 | ⭐⭐⭐⭐⭐ | ~1 MB |

ASTC 6x6 | 同上 | ⭐⭐⭐⭐ | ~0.45 MB |

ETC2 | 全安卓(OpenGL ES 3.0+) | ⭐⭐⭐ | ~0.5 MB |

RGBA 未压缩 | — | 原始 | ~4 MB |

⚡ 对性能的影响

未压缩 vs ASTC → 内存占用差 4~8 倍

内存爆了 → 闪退（OOM），这是玩家最不可接受的体验

包体过大 → 下载转化率下降（每增加 100MB，下载流失约 10-15%）

错误的压缩格式 → GPU 要实时解压，吃 CPU + 加载变慢

📦 包体红线：立项时必须定包体预算（如初装 ≤1.5GB），贴图通常占 60-70%

💥 最大风险：美术交付 PSD 但没配好压缩格式 → 上线后闪退，这是 P0 事故

🚨 风险信号：TA 说"某些贴图没走压缩"或"内存超了要重做贴图"，意味着美术返工 + 排期后延

📊 你可以这样问："目前项目统一用的什么压缩格式？有没有未压缩的贴图残留？"

🦴 骨骼数与蒙皮权重

Bone Count & Skin Weight（骨骼绑定 & 顶点权重）

CPU 开销
内存占用

🔍 一句话解释

骨骼就是角色身体里的"骨架"，驱动模型动起来。蒙皮权重是告诉每个顶点"你听几根骨骼的话、各听多少"。骨骼越多，动画越细腻，但 CPU 每帧要算的矩阵运算就越多。

⚡ 对性能的影响

每根骨骼每帧需要做矩阵变换运算 → 骨骼越多 CPU 越累

蒙皮权重 4 骨影响（4 bones per vertex）是移动端标准

同屏多角色时骨骼开销线性叠加（10个角色×80骨骼=800矩阵计算）

骨骼还影响动画文件大小和加载速度

🔧 常见优化手段

LOD 骨骼：远处角色自动减少骨骼（如 80→30→15）

GPU Skinning：把蒙皮计算搬到 GPU（释放 CPU）

Animation Instancing：相同动作的 NPC 共享计算

区分主角/NPC/怪物：只有主角用高精度骨骼

👥 关注同屏人数：策划想同屏 20 个角色，你得和程序确认骨骼预算是否够

💃 关注动作需求：策划想要"披风飘动 + 头发物理"，每个都要额外骨骼链，会显著增加开销

🚨 风险信号：动作组说"加几根辅助骨没什么大碍" → 必须让 TA 评估总量

📊 你可以这样问："主角和 NPC 各用多少根骨骼？同屏最大角色数测试过吗？"

💾 纹理内存 (Texture Memory)

Texture Memory / VRAM — 贴图显存占用

内存杀手
闪退元凶
包体大户

🔍 一句话解释

所有贴图（角色皮肤、场景地面、UI 图片、特效火焰…）加载到内存里占的空间就是纹理内存。它通常是游戏内存的最大头（占 50-70%）。移动端没有独立显存，纹理直接吃系统内存。

⚡ 对性能的影响

一张 2048×2048 RGBA 未压缩 = 16MB

ASTC 4x4 压缩后 ≈ 4MB，但加上 Mipmap 还要 ×1.33

iOS 2GB 设备内存警戒线约 900MB，超了就闪退

安卓碎片化严重，1.5GB 机型大量存活

🔧 常见优化手段

贴图降分辨率：2048→1024 = 内存直接砍 75%

Mipmap：远处用小贴图（+33% 内存换巨大性能提升）

Texture Streaming：按需加载/卸载贴图

通道打包：4 张灰度图合成 1 张 RGBA（省 3 张贴图的开销）

UI 图集：碎图合并为大图集，减少内存碎片

💥 第一风险：闪退 = 流失。纹理内存超标是移动端闪退的 No.1 原因

📱 机型分级：必须建立高/中/低三档画质，低端机用低分辨率贴图

🚨 风险信号：QA 报"特定场景闪退"、"切换角色后闪退" → 大概率是纹理内存峰值超标

📊 你可以这样问："当前版本在 iPhone 12 上的内存峰值是多少？贴图占了多少？"

3️⃣ 场景化红线：3 种典型游戏品类

不同品类的性能瓶颈截然不同。以下给出 3 种中型移动游戏的典型预算分配。

🎴 二次元卡牌 / 回合制 RPG

核心瓶颈：角色渲染品质 > 性能 — 玩家看角色立绘和 Live2D，对帧率容忍度高

重点项 | 参考值 | 说明 |

主角面数 | 1.5~2 万 | 立绘级品质，允许稍高 |

同屏角色 | 6~8 个 | 3v3 或 4v4 对战 |

帧率目标 | 30fps | 回合制对帧率不敏感 |

核心风险 | 卡牌立绘贴图数量爆炸（100+角色 × 多套皮肤），包体和内存是主要战场 |

⚔️ ARPG / 动作游戏

核心瓶颈：帧率稳定性 — 打击感需要稳定 30fps（理想 60fps），掉帧 = 手感崩塌

重点项 | 参考值 | 说明 |

主角面数 | 1~1.5 万 | 需要留空间给特效 |

同屏怪物 | 10~20 个 | 割草类可达 30+，需严控单体面数 |

帧率目标 | 稳 30fps / 高端 60fps | 帧率波动 ≤5fps |

特效预算 | 同屏粒子 ≤500 | 大招 Overdraw 控制在 3x 以内 |

核心风险 | 特效 Overdraw 叠加 + 同屏大量怪物骨骼 → CPU/GPU 双瓶颈 |


核心瓶颈：场景复杂度 + 内存管理 — 大世界 = 海量资源动态加载/卸载

重点项 | 参考值 | 说明 |

可视范围面数 | 15~20 万（含 LOD） | 远处 LOD 大幅减面 |

LOD 层级 | 至少 3 级 | LOD0/1/2 + HLOD |

Texture Streaming | 必须启用 | 否则内存秒爆 |

📦

包体增量 — "这个版本包体增加了多少 MB？主要是什么资源导致的？"

🖼️

贴图审查 — "有没有发现未压缩的贴图或超规格的贴图？CI 的自动检查通过了吗？"

🦴

骨骼用量 — "新上线的角色骨骼数符合规范吗？有没有超标的？"

🔥

发热与功耗 — "在目标设备上连续游玩 30 分钟，温度和降频情况怎么样？"

5️⃣ FAQ：项目经理最常问的 10 个问题

❓ Q1：策划说要同屏 50 个怪，我怎么判断可不可行？

快速估算公式：50 怪 × 3000面/怪 = 15万面，已经超红线。再加上50 × 30骨 = 1500 骨骼矩阵，CPU 也扛不住。对策：1) 远处怪物用 Billboard 替代；2) 近处只渲染 10~15 个精细模型；3) 其余用 2D 特效 fake。

❓ Q2：美术说"这个效果必须用 2048 贴图"，我怎么挡？

问两个问题：1) "这张贴图在手机屏幕上实际显示多大？" —— 如果角色只占屏幕 1/4，2048 和 1024 的区别肉眼几乎看不出；2) "目前项目总的贴图内存预算还剩多少？" 拿数字说话比感觉管用。

❓ Q3：什么是"合批率"？达标线是多少？

合批率 = (1 - 实际Drawcall / 未优化Drawcall) × 100%。通常合批率≥60% 是健康的。低于 40% 说明材质混乱严重，需要 TA 介入整理。

❓ Q4："Shader 变体爆炸"是什么意思？会影响什么？

一个 Shader 配上不同的开关组合（有无法线、有无自发光、有无透明…），每种组合编译出一个"变体"。变体数指数增长（10个开关 = 1024 种）。影响：包体暴涨 + 首次加载卡顿。让 TA 定期用 Shader Variants Collection 工具清理。

❓ Q5：特效说"就加了一层 Bloom"，为什么帧率掉了 5 帧？

Bloom 是全屏后处理特效，需要多次 Downsample + Upsample + 混合，每次都是全屏 Overdraw。在低端机上一个 Bloom 就可以吃掉 3~5ms。建议：Bloom 降低迭代次数、降低后处理分辨率（如半分辨率 Bloom）。

❓ Q6：什么时候该上 LOD？LOD 的工作量大吗？

所有 3D 场景物件和非主角角色都应该有 LOD。工作量：如果项目用 UE5 Nanite 可自动生成；Unity 可用 Simplygon/InstaLOD 插件半自动生成。手工 LOD 通常额外增加20-30% 工期，但对性能的回报是 50% 以上。

❓ Q7：QA 说"低端机偶尔闪退"，怎么快速定位是不是美术资源的锅？

让程序查 Crash Log 里是否有 OOM / Memory Warning / jetsam 关键字。如果有 → 99% 是纹理内存超标。接下来让 TA 用 Memory Profiler 抓内存快照，看哪些贴图在内存里但当前场景根本不需要（资源未卸载）。

❓ Q8：安卓和 iOS 的性能差异大吗？需要分别测试吗？

必须分别测。iOS Metal 渲染效率高但内存限制严（2GB 设备 OOM 阈值约 900MB）；安卓碎片化严重，GPU 品牌多（Adreno/Mali/PowerVR），Shader 兼容性可能不同。建议至少覆盖3 台 iOS + 5 台安卓做基准测试。

❓ Q9：如何设定合理的性能优化里程碑？

推荐 "三次性能审查" 节奏：
• Alpha：确认基准帧率、内存上限，建立自动化监控
• Beta：全场景全流程真机测试，输出性能热点报告
• RC：冻结资源规格，只允许优化不允许新增高消耗资源

❓ Q10：我不懂技术，但我想监控性能状况，有什么工具推荐？

工具 | 平台 | 适合谁 | 用途 |

PerfDog | iOS/Android | ⭐ APM 首推 | 无需代码，USB 连手机即可看帧率/内存/CPU/GPU |

UPR (Unity) | Unity | 程序/TA | Unity 官方性能分析云平台 |

Unreal Insights | UE5 | 程序/TA | UE5 内置全链路性能分析 |

Xcode Instruments | iOS | 程序/TA | Apple 官方，内存/GPU 分析精准 |

RenderDoc | 全平台 | TA | 逐帧 GPU 分析，查 Drawcall/Overdraw |

建议：要求 TA 每周用 PerfDog 出一份一页纸报告（帧率/内存/温度曲线），发到项目群。
