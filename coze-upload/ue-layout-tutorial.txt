  📦 1. 布局面板（Panel）详解与对比
  在 UMG 中，面板决定了子元素如何排列。选择正确的面板是 UI 拼接的第一步。

  1.1 画布面板 (Canvas Panel) vs 覆层 (Overlay)
  这是初学者最容易混淆的两个容器。它们都允许放置多个子控件，但运作逻辑完全不同。
  
  
    
      🖼️ 画布面板 (Canvas Panel)
      作用： 允许你在其中任意拖拽摆放控件。子控件依靠 Anchor (锚点) 进行定位。
      
        定位方式： 通过 Position X/Y 和 Size X/Y 控制位置和大小。
        适用场景： 界面最底层的根容器；需要固定在屏幕特定角落（如右上角小地图、右下角技能）的零散元素。
        缺点： 性能开销略大。如果在复杂的 UI 内部频繁使用（比如一个按钮内部还套一层 Canvas），会导致极大的性能浪费。
      
    
    
      🥞 覆层 (Overlay)
      作用： 将多个控件像三明治一样“叠”在一起。最先放入的在底层，后放入的在上层。
      
        定位方式： 没有锚点和 X/Y 坐标。只能通过 Padding (边距) 和 Alignment (水平/垂直对齐，如居左、居中、拉伸) 来排布。
        适用场景： 按钮组件的内部（背景图叠上文字）；装备格（底图叠上图标、再叠上边框）；任何需要背景图和内容叠放的地方。
        优点： 性能极好！官方强烈推荐：能用 Overlay 解决的层叠，绝对不要用 Canvas Panel。
      
    
  

  1.2 线性布局：Horizontal Box vs Vertical Box
  这两种盒子用于让控件自动排队，无需手动计算坐标。
  
    面板名称作用与效果核心参数说明
    
      Horizontal Box水平框
      子控件从左到右依次排列。
      
        当控件放入 Box 中，它会多出一个 Slot (插槽) 属性面板：
        • Size: 
           - Auto：控件按自身内容大小显示（如字多长就多宽）。
           - Fill：填满剩余空间。多个控件选 Fill，可通过填入权重数字（如 1和2）按比例瓜分宽度。
      
    
    
      Vertical Box垂直框
      子控件从上到下依次排列。
      与水平框相同，通常用于排列菜单列表、选项按钮等。同样依靠 Padding 控制条目间的间距。
    
  

  1.3 高级限制容器
  用于特殊的大小限制与缩放需求。
  
    尺寸框 (Size Box)： 强行指定内部控件的宽度和高度，或设置最小/最大宽高度。比如：限制输入框宽度最少 200px。
    缩放框 (Scale Box)： 内部元素超出指定尺寸时，自动按比例缩小（或放大）以适应空间。非常适合处理“多语言文本过长”的自动缩放。
    安全区 (Safe Zone)： 极其重要！放在整个界面的最外层（或者在底层 Canvas 内部套一层）。勾选左右边距后，能自动避开手机的“刘海屏”或“打孔屏”区域，防止 UI 被遮挡。
  




  📜 2. 列表与滚动控件全解析
  当内容超过屏幕时，我们需要滚动功能。UMG 提供了三种不同级别的控件。

  2.1 Scroll Box（滚动框） vs Wrap Box（自动换行框）
  
    
      ↕️ Scroll Box (滚动框)
      最基础的滚动视图。只要把控件丢进去，超出范围就会自动出现滚动条。
      
        特点： 简单粗暴，有多少条目就渲染多少个节点。
        适用场景： 内容固定、条目较少（如只有 10 条的设置面板、长篇的规则说明文本）。
        致命缺点： 如果列表里有 1000 条数据，它会同时渲染 1000 个 UI 实例，瞬间卡死游戏。
      
    
    
      ↩️ Wrap Box (自动换行框)
      这不是滚动视图，它只是一个排版容器。子元素从左向右排，排满一行后自动掉到下一行。
      
        特点： 没有滚动条，只是控制流式布局。
        适用场景： 技能标签词云、长度不一的道具掉落图标展示。
        注意： 如果排下去了还是超出屏幕，需要把 Wrap Box 整体放进一个 Scroll Box 里才能滚动。
      
    
  

  2.2 性能救星：List View 与 Tile View（数据驱动列表）
  针对拥有成百上千条数据的界面（如排行榜、大背包），必须使用这两种视图。
  
    控件视觉表现核心原理与配置方法
    
      List View列表视图
      单列垂直往下排队滚动。
      
        原理： 采用“对象池复用”技术。屏幕上最多能显示 10 个条目，引擎就只创建 12 个 UI 实例，滑动时上下循环利用，性能极佳。
        配置方法：
        1. 必须单独创建一个 UMG 蓝图作为“单条子项”，该蓝图需继承 User Object List Entry 接口。
        2. 在 List/Tile View 的 Entry Widget Class 参数中，选择刚才创建的子项蓝图。
        3. 不能手动拖拽 UI 进去，必须通过蓝图代码 Add Item 传入数据对象。
      
    
    
      Tile View瓦片视图
      网格状排队（像九宫格），一行排满再排下一行。
    
  
  
    💡 Tile View 专属重要参数：
    • Entry Height / Entry Width：定义每个“瓦片（格子）”的固定尺寸。它决定了一行能排下多少个。
    • Tile Alignment：当最后一行不满时，是居左、居中还是分散对齐。
  




  🔤 3. 基础与交互组件详解
  
  3.1 图片 (Image) 的切图设置
  UI 拼接中最常用的是九宫格（Slice / Box）以保证边框拉伸不形变。
  
    Brush -> Draw As： 默认是 Image（直接拉伸）。改为 Box（即九宫格模式）。
    Margin (边距)： 当选为 Box 时，需要设置 Margin。该值通常是一个小数（如 0.25），代表相对整张图片的比例；或者在材质/贴图设置中填入绝对像素值。
  

  3.2 玩家文本输入：Editable Text 组件
  当需要玩家在界面上打字（如输入名字、聊天、密码）时，必须使用输入框组件。
  
  
    
      🔡 可编辑文本 (Editable Text)
      用于单行输入（如注册账号、兑换码）。不支持换行。
      
        Hint Text： 提示文案（如“请输入玩家姓名…”）。输入内容后自动消失。
        Hint Text Color： 提示文案的颜色，通常设为半透明灰色。
        Is Password： 勾选后，输入的内容会变成星号 ***。
        Is Read Only： 勾选后，框存在但禁止输入（通常用作可复制但不可改的展示框）。
        Appearance (外观)： 可以在这里设置输入文字的字体、大小和颜色。
      
    
    
      🔠 多行可编辑文本 (Multi-Line Editable Text)
      用于大段输入（如意见反馈、公会宣言）。
      
        Auto Wrap Text： 强烈建议勾选。当玩家输入到边界时自动换行。
        参数基本与单行框类似，但不支持 Is Password（没人会写多行密码）。
        注意： 默认情况下背景是完全透明的，你通常需要在它下面垫一张 Image 作为输入框底图，再用 Overlay 把它们包起来。
      
    
  

  3.3 文本 (Text) 与 富文本 (Rich Text Block)
  
    Text (普通文本)： 只有一种颜色、一种字体。参数 Justification 控制左/中/右对齐。
    Rich Text Block (富文本)： 允许在一句话里改变特定文字的颜色、大小，甚至插入图片（如图标表情）。
      用法：需要在项目里创建 Data Table（数据表）定义样式，然后输入 <Red>红色文字</> 或者 <img id="GoldIcon"/> 这样的标签。
    
  




  🎯 4. 核心实操：锚点、自适应与层级搭建
  了解完节点后，如何组合成一个不随屏幕分辨率变形的界面？

  4.1 第一法则：打死定在原地的“锚点 (Anchor)”
  在 Canvas Panel 下的子控件，它的位置是相对于“锚点花（那个像小菊花一样的图标）”来计算的。
  
    界面左上角的返回按钮： 锚点必须选为 左上角。
    界面右下角的开战按钮： 锚点必须选为 右下角。否则在带鱼屏手机上，按钮会飘到屏幕中间。
    全屏背景图： 锚点选为 四边拉伸 (右下角的方形框)，此时原有的尺寸面板会变成 Offsets (Left, Right, Top, Bottom)，把这四个值全部归 0，背景就会死死贴住屏幕边缘。
    居中弹窗： 锚点选 中心点。同时将 Alignment (对齐) 设为 X:0.5, Y:0.5。这是因为默认坐标是以控件的左上角计算的，改 Alignment 可以让弹窗的真正中心点与屏幕中心对齐。
  

  4.2 标准面板搭建结构推荐
  一个标准的、性能好的系统界面，它的层级树（Hierarchy）应该是这样的：
  
[根目录] UserWidget
 └── Canvas Panel (界面最底层画板)
      ├── Image (全屏背景图，锚点四边拉伸，Offset全为0)
      └── Safe Zone (安全区，勾选Pad四边，适配异形屏)
           └── Overlay (覆层，铺满安全区)
                ├── Image (通用弹窗大底板图，剧中对齐)
                ├── Vertical Box (左中右布局中的某个列)
                │    ├── Text (标题)
                │    └── List View (核心道具列表，设为Fill填满剩余)
                └── Button (右上角关闭按钮，由于在Overlay里，设为 右上角对齐)
  

  
    💡 终极实操排错指南：
    ❓ 按钮点不到？ 检查按钮上方是不是有一个透明的 Image 背景图覆盖了它。将其 Visibility 从 Visible 改为 Self Hit Test Invisible（自身不挡射线）。
    ❓ 列表只能看不能滑？ 检查 Scroll Box 或者 List View 外部是否被设定了固定大小但没被正确约束。或者内部控件占用的区域太小。
    ❓ 文字被截断？ 检查 Text 组件是否放入了 Size Box 并且尺寸被限制得太死，或者可以勾选 Auto Wrap Text。
